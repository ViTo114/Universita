# INTRODUZIONE AD API GATEWAT A AWS LAMBDA
Nell'ecosistema del cloud computing moderno, la capacità di costruire applicazioni scalabili, resilienti e a basso costo di gestione è di fondamentale importanza. Al centro di questa rivoluzione architetturale si trovano due servizi cloud fondamentali offerti da Amazon Web Services (AWS): API Gateway e Lambda. La loro combinazione strategica permette agli sviluppatori di creare potenti applicazioni serverless, spostando il focus dalla gestione dell'infrastruttura alla scrittura della logica di business. Comprendere il ruolo e il funzionamento di ciascuno di questi servizi è il primo passo per padroneggiare lo sviluppo di sistemi informativi di nuova generazione.

AWS API Gateway è un servizio completamente gestito che agisce come punto di ingresso unificato ("front door") per le applicazioni, consentendo la creazione, la pubblicazione, la manutenzione e la protezione di interfacce di programmazione delle applicazioni (API) su qualsiasi scala. Supporta diversi tipi di API, tra cui REST, HTTP e WebSocket, e si integra nativamente con un'ampia gamma di servizi AWS. Funge da intermediario per carichi di lavoro in esecuzione su istanze EC2, per la logica di calcolo contenuta in funzioni AWS Lambda, per applicazioni web e per applicazioni di comunicazione in tempo reale. Quando si crea un'API RESTful tramite API Gateway, si definisce un'interfaccia basata sul protocollo HTTP che implementa metodi standard come GET, POST e DELETE. Questo approccio favorisce una comunicazione client-server di tipo _stateless_, dove ogni richiesta del client contiene tutte le informazioni necessarie per essere elaborata, garantendo scalabilità e disaccoppiamento.

Parallelamente, AWS Lambda rappresenta il cuore pulsante dell'informatica serverless. Il suo scopo principale è permettere l'esecuzione di codice senza doversi preoccupare della gestione dei server sottostanti. Questo approccio offre vantaggi significativi: AWS gestisce in modo autonomo tutte le attività operative, dalla manutenzione dei server agli aggiornamenti del sistema operativo, fino al provisioning della capacità e alla configurazione del logging. Le funzioni Lambda vengono eseguite solo quando sono effettivamente necessarie, scalando automaticamente per gestire un numero variabile di richieste, da poche al giorno a migliaia al secondo. Il modello di pagamento riflette questa efficienza: si paga solo per il tempo di calcolo effettivamente consumato (pay-per-use), eliminando i costi legati ai server inattivi.

Dopo aver analizzato le caratteristiche di questi due potenti servizi, è naturale chiedersi come possano essere combinati per creare architetture serverless efficienti. L'integrazione tra API Gateway e Lambda non è solo possibile, ma costituisce uno dei pattern architetturali più diffusi e robusti nel cloud, che esploreremo nel dettaglio nel capitolo successivo.




# L'ARCHITETTURA SERVELESS: INTEGRAZIONE TRA API GATEWAY E LAMBDA
Il concetto di architettura serverless per una REST API trova la sua espressione più comune e potente nell'integrazione tra API Gateway e Lambda, che ne costituisce il modello fondamentale. Questo pattern è diventato uno standard de facto per lo sviluppo di applicazioni agili, poiché permette di esporre logica di business complessa tramite semplici endpoint HTTP, senza i costi e la complessità legati alla gestione di un'infrastruttura server tradizionale. L'importanza di questo modello risiede nella sua capacità di ridurre il time-to-market e i costi operativi, consentendo ai team di concentrarsi esclusivamente sulla logica applicativa.

Il flusso di una richiesta in un'architettura di questo tipo è elegante nella sua semplicità e si articola in una sequenza ben definita di interazioni tra il client, API Gateway e la funzione Lambda. Il processo inizia quando un client, come un'applicazione web o mobile, invia una richiesta HTTP a un endpoint definito in API Gateway. A questo punto, l'API Gateway riceve la richiesta e, in base alla sua configurazione, la instrada alla funzione Lambda designata come backend. Questo è il primo passaggio cruciale, in cui l'API Gateway agisce da intermediario e punto di controllo. Questa separazione tra il punto di ingresso (API Gateway) e la logica di calcolo (Lambda) è un principio chiave delle architetture a microservizi, in quanto consente a ciascun componente di scalare e di essere aggiornato in modo indipendente.

Una volta invocata, la funzione Lambda esegue il codice al suo interno, elaborando i dati contenuti nella richiesta. La logica della funzione può eseguire qualsiasi operazione, come calcoli, interrogazioni a un database o interazioni con altri servizi. Al termine dell'elaborazione, la funzione costruisce e restituisce una risposta, che viene inviata nuovamente all'API Gateway. Questo costituisce il secondo passaggio del flusso, in cui la logica di business viene effettivamente eseguita.

Infine, nel terzo e ultimo passaggio, l'API Gateway riceve la risposta dalla funzione Lambda e la inoltra al client originale che ha avviato la comunicazione. L'API Gateway può anche trasformare la risposta, se necessario, prima di inviarla, completando così il ciclo richiesta-risposta in modo completamente gestito e scalabile. Il diagramma concettuale, che visualizza il flusso `Clients -> REST API -> Lambda function`, rafforza questa comprensione, mostrando chiaramente come l'API Gateway funga da ponte tra il mondo esterno e la logica di calcolo serverless.

Dopo aver compreso la teoria alla base di questa potente architettura, il passo successivo è imparare come implementarla e testarla efficacemente. Per fare ciò, è indispensabile disporre di un ambiente di sviluppo locale che possa simulare l'ecosistema AWS, ed è qui che entra in gioco uno strumento fondamentale come LocalStack.

Questa distinzione consente agli sviluppatori di scegliere il livello di emulazione più adatto alle proprie esigenze.

Il primo passo pratico per iniziare a lavorare con questo ambiente è l'avvio del container LocalStack. Questa operazione viene tipicamente eseguita tramite Docker o altri metodi di containerizzazione preferiti. L'avvio dell'ambiente LocalStack è il prerequisito fondamentale per tutte le operazioni successive, poiché crea l'istanza locale dei servizi AWS con cui andremo a interagire tramite la riga di comando o gli SDK.

Una volta che l'ambiente LocalStack è in esecuzione, il primo componente da creare per la nostra architettura serverless è la logica di business. # CREAZIONE DI UNA FUNZIONE LAMBDA
Con l'ambiente di sviluppo locale pronto, possiamo ora concentrarci sulla creazione del "cervello" della nostra applicazione serverless: la funzione Lambda. Questo capitolo fornisce una guida dettagliata per scrivere il codice della funzione, comprendere la sua struttura interna e prepararla per il deploy nell'ambiente LocalStack. Questo processo è fondamentale, poiché la funzione conterrà la logica applicativa che verrà eseguita ogni volta che la nostra API riceverà una richiesta.

Il primo passo consiste nella scrittura del codice, che verrà salvato in un file denominato `lambda.js`. La logica implementata nella funzione di esempio, `apiHandler`, è progettata specificamente per gestire richieste di tipo GET. Il suo compito è estrarre i parametri `name` e `surname` dalla query string dell'URL della richiesta in arrivo. Successivamente, utilizza questi parametri per costruire un messaggio di saluto personalizzato, che viene poi formattato in una risposta JSON standard, completa di codice di stato HTTP 200.

Per comprendere appieno il funzionamento del codice, è essenziale analizzare il concetto di "handler" e i parametri che riceve. L'handler, in questo caso identificato dalla stringa `lambda.apiHandler`, rappresenta il punto di ingresso che il servizio Lambda invoca per eseguire la funzione. Il runtime di Lambda passa a questo metodo tre argomenti principali. Il primo è l'oggetto `event`, che contiene tutti i dati e i metadati inviati dall'invoker; la sua struttura dipende da quale servizio AWS invoca la funzione Lambda. Il secondo è l'oggetto `context`, che fornisce informazioni di runtime sull'invocazione, sulla funzione stessa e sull'ambiente di esecuzione. Infine, il terzo argomento è la funzione `callback`, che viene utilizzata negli handler non asincroni per restituire la risposta al chiamante, segnalando il completamento dell'esecuzione.

Una volta scritto il codice, è necessario prepararlo per il caricamento su LocalStack. Questo processo richiede la creazione di un pacchetto di deploy, che è semplicemente un file compresso `.zip` contenente il file `lambda.js` e le sue eventuali dipendenze. Successivamente, si utilizza il comando `awslocal lambda create-function` per caricare la funzione. Questo comando richiede diversi parametri chiave per configurarla correttamente: `--function-name` per assegnare un nome univoco (nel nostro esempio, `apigw-lambda`), `--handler` per specificare il punto di ingresso del codice, `--runtime` per definire l'ambiente di esecuzione (es. `nodejs16.x`), `--memory-size` per allocare la quantità di memoria, `--zip-file` per indicare il percorso del pacchetto di deploy e `--role` per assegnare un ruolo di esecuzione con i permessi necessari. La definizione di questi parametri è centrale nel modello serverless, poiché influenza direttamente sia le performance che i costi della funzione, legando le risorse computazionali strettamente ai requisiti del codice.

Con la funzione Lambda correttamente creata e caricata nel nostro ambiente locale, la logica di business è pronta. Il passo successivo consiste nel costruire il punto di accesso pubblico che permetterà ai client di invocare questa logica: la creazione e la configurazione della REST API, che sarà l'argomento del prossimo capitolo.



# CONFIGURAZIONE DELLA REST API IN API GATEWAY
Ora che la nostra logica di business è incapsulata in una funzione Lambda, è necessario esporla al mondo esterno tramite un endpoint pubblico. In questa fase, utilizzeremo API Gateway per configurare tale endpoint, definendo la struttura gerarchica dell'API, le sue risorse e i metodi HTTP che i client potranno invocare. Questo processo trasforma la nostra funzione in un servizio web a tutti gli effetti.

Il primo passo consiste nella creazione della REST API di base, che funge da contenitore per tutte le risorse e i metodi. Questo si ottiene tramite il comando `awslocal apigateway create-rest-api`. È di fondamentale importanza prendere nota dell'attributo `id` restituito da questo comando, poiché questo identificatore univoco sarà richiesto in tutti i comandi successivi per specificare a quale API ci stiamo riferendo.

Successivamente, dobbiamo gestire le "risorse" dell'API, che rappresentano le entità logiche accessibili tramite percorsi URL. Ogni API ha una risorsa radice (`/`), il cui ID può essere ottenuto con il comando `get-resources`. Partendo da questa radice, possiamo creare nuove risorse. Ad esempio, per creare una risorsa accessibile al percorso `/hello`, si utilizza il comando `create-resource`. Questo comando richiede l'ID della risorsa genitore (in questo caso, la radice) e il parametro `--path-part`, che definisce l'ultimo segmento del percorso URL per la nuova risorsa, in questo caso `"hello"`.

Una volta definita la risorsa, il passo successivo è associare ad essa un metodo HTTP. Per il nostro esempio, collegheremo il metodo `GET` alla risorsa `/hello` utilizzando il comando `put-method`. Questo comando richiede diversi parametri importanti per la sua configurazione. Il parametro `--http-method` specifica il verbo HTTP (`GET`), mentre `--authorization-type` definisce il meccanismo di autorizzazione; il valore `NONE` indica che l'endpoint è accessibile pubblicamente senza autenticazione. Un parametro cruciale è `--request-parameters`, che definisce una mappa chiave-valore per i parametri attesi. La chiave deve seguire il pattern `method.request.{location}.{name}`, dove `location` può essere `querystring`, `path` o `header`, e `name` è il nome del parametro. Il valore associato a ciascuna chiave è un booleano (`true` o `false`) che indica se il parametro è, rispettivamente, obbligatorio o opzionale.

A questo punto, la struttura dell'API è definita, con una risorsa e un metodo pronti ad accettare richieste. Tuttavia, il metodo `GET` non è ancora collegato a nessuna logica di backend. L'ultimo passo cruciale, che verrà trattato nel capitolo successivo, è quindi quello di integrare questo metodo con la funzione Lambda che abbiamo precedentemente creato, per completare il flusso di comunicazione.




# INTEGRAZIONE, DEPLOY E TEST DELLA REST API
Siamo giunti alla fase finale dell'implementazione, in cui tutti i componenti precedentemente creati vengono connessi e resi operativi. In questo capitolo, collegheremo il metodo `GET` della nostra API alla funzione Lambda, distribuiremo la configurazione dell'API per renderla accessibile pubblicamente e, infine, la testeremo per verificare che l'intera architettura funzioni come previsto.

L'integrazione tra il metodo dell'API e la funzione Lambda si realizza tramite il comando `put-integration`. Un aspetto fondamentale di questa operazione è la scelta del tipo di integrazione. Utilizzando il valore `AWS_PROXY`, si abilita la cosiddetta _Lambda Proxy Integration_, una modalità che semplifica notevolmente la creazione di API serverless. Con questa configurazione, API Gateway inoltra l'intera richiesta HTTP ricevuta dal client direttamente alla funzione Lambda, senza alcuna trasformazione. Allo stesso modo, la risposta prodotta dalla Lambda viene restituita così com'è al client. Tra gli altri parametri del comando, `--passthrough-behavior` specifica come gestire le richieste senza template di mappatura, mentre `--integration-http-method` definisce il metodo HTTP che API Gateway usa per invocare il backend; per l'integrazione con Lambda, questo valore deve essere `POST`.

Un'attenzione particolare merita la struttura dell'argomento `--uri` fornito al comando `put-integration`. Questo URI segue un pattern ARN (Amazon Resource Name) specifico per l'integrazione con Lambda: `arn:aws:apigateway:{region}:lambda:path/2015-03-31/functions/{lambda_arn}/invocations`. Ogni parte di questo ARN ha un significato preciso: `arn:aws:apigateway:{region}:lambda` è il prefisso che identifica il servizio, la regione e l'integrazione; `path/2015-03-31` indica la versione dell'integrazione, stabile dal 31 marzo 2015; `functions/{lambda_arn}` specifica l'ARN completo della funzione Lambda da invocare; infine, il suffisso statico `invocations` istruisce API Gateway a chiamare l'API di invocazione di Lambda.

Una volta configurata l'integrazione, l'API deve essere distribuita. Il "deployment" è l'azione che pubblica una specifica configurazione dell'API, rendendola richiamabile tramite un URL pubblico. Questo si ottiene con il comando `create-deployment`, che richiede l'ID dell'API (`--rest-api-id`) e uno `--stage-name`, ovvero un nome per lo stage di deploy (ad esempio `test` o `prod`), che permette di gestire diverse versioni della stessa API.

Con l'API finalmente distribuita, è possibile procedere al test. Esistono diversi formati di URL per invocare l'endpoint in un ambiente LocalStack, che sono metodi alternativi per ottenere lo stesso risultato. I formati raccomandati sono: `http://localhost:4566/restapis/{REST_API_ID}/{STAGE}/_user_request_/{PATH}?{KEY-VALUE-PAIRS}`, `http://{REST_API_ID}.execute-api.localhost.localstack.cloud:4566/{STAGE}/{PATH}` e `http://localhost:4566/_aws/execute-api/{REST_API_ID}/{STAGE}/{PATH}`. Utilizzando l'URL specifico per il nostro esempio, come `http://localhost:4566/restapis/cor3o5oeci/test/_user_request_/hello?name=Mario&surname=Rossi`, dovremmo ricevere la risposta JSON attesa, `{"message": "Hello from Lambda: Mario Rossi"}`, a conferma del corretto funzionamento dell'intero sistema.

Avendo completato con successo la creazione e il test di un'API con metodo GET, siamo ora pronti a consolidare le nostre competenze e a esplorare scenari più complessi e funzionalità aggiuntive attraverso una serie di esercizi pratici.




# ESERCITAZIONI PRATICHE E INTEGRAZIONE CON S3
L'applicazione pratica dei concetti teorici è un passaggio fondamentale per consolidare l'apprendimento e acquisire una reale padronanza delle tecnologie. In questo capitolo finale, verranno proposti due esercizi pratici che estendono le competenze acquisite finora, introducendo l'uso del metodo HTTP POST e l'integrazione con un altro servizio AWS di primaria importanza: Amazon S3 (Simple Storage Service).

Per consolidare la comprensione delle API, si andrà a implementare una delle operazioni più comuni: la creazione di dati tramite il metodo POST, che gestisce i dati inviati nel corpo della richiesta. L'obiettivo è aggiungere un metodo `POST` alla risorsa `/hello`. Questo nuovo metodo dovrà essere configurato per accettare un corpo della richiesta in formato `application/json`, contenente i campi `name` e `surname`. Di conseguenza, sarà necessario creare o modificare una funzione Lambda associata, in modo che sia in grado di leggere e processare il corpo della richiesta per poi restituire un messaggio di saluto formattato. È importante ricordare che, dopo ogni modifica alla configurazione dell'API, è necessario effettuare nuovamente il deploy per rendere le modifiche attive.

Questo secondo esercizio introduce un pattern architetturale fondamentale: l'integrazione asincrona tra servizi AWS, dimostrando come una funzione Lambda possa agire da orchestratore per archiviare dati in un servizio di storage persistente come S3. L'obiettivo è creare una nuova risorsa, `/save`, che risponda a richieste `POST`. La funzione Lambda collegata a questa risorsa dovrà estrarre il corpo della richiesta, contenente sempre i campi `name` e `surname`, e utilizzarlo per salvare un nuovo oggetto all'interno di un bucket S3 precedentemente creato e denominato `test-s3-bucket`. Il nome del file da salvare dovrà seguire un formato specifico, ovvero `person-{name}-{surname}.txt`.

Per facilitare lo sviluppo della funzione Lambda per il secondo esercizio, vengono forniti alcuni suggerimenti di codice. È necessario innanzitutto importare l'AWS SDK per JavaScript per poter interagire con S3. La funzione handler dovrebbe essere dichiarata come `async` per gestire correttamente le operazioni asincrone. All'interno della funzione, si dovrà effettuare il parsing della stringa JSON contenuta in `event.body` (ad esempio, tramite `JSON.parse(event.body)`) per estrarre i dati e, infine, utilizzare il metodo `s3.putObject` per eseguire l'operazione di salvataggio dei dati nel bucket S3 specificato.

Le competenze acquisite attraverso questi esercizi, che spaziano dalla creazione di una semplice API GET all'implementazione di logiche più complesse con metodi POST e integrazioni con servizi di storage come S3, forniscono una solida e indispensabile base per affrontare lo sviluppo di applicazioni serverless sofisticate e robuste all'interno del vasto ecosistema di Amazon Web Services.
