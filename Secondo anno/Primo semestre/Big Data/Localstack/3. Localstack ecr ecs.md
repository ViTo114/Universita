# INTRODUZIONE AD AMAZON ELASTIC CONTAINER REGISTRY (ECR)
Nello sviluppo di applicazioni moderne, disporre di un registro di container affidabile è un requisito fondamentale. Questo componente infrastrutturale non solo archivia le immagini delle applicazioni, ma garantisce anche che siano accessibili, sicure e gestibili durante tutto il ciclo di vita dello sviluppo. Amazon Elastic Container Registry (ECR) si inserisce in questo contesto come un servizio completamente gestito che risolve questa esigenza, offrendo una profonda integrazione con l'ecosistema AWS e semplificando la gestione delle immagini container.

**Amazon Elastic Container Registry (ECR)** è un servizio completamente gestito per l'archiviazione e la gestione di immagini container Docker. La sua natura gestita elimina la complessità operativa legata alla manutenzione di un registro privato, permettendo agli sviluppatori di concentrarsi sulla creazione delle applicazioni.

Le caratteristiche chiave di ECR ne evidenziano la potenza e la flessibilità. Si integra nativamente con altri servizi AWS come Lambda, Elastic Container Service (ECS) ed Elastic Kubernetes Service (EKS), semplificando i flussi di deployment. Per impostazione predefinita, i repository creati in ECR sono privati, garantendo un elevato standard di sicurezza, sebbene sia possibile crearne anche di pubblici. Inoltre, ECR è un servizio regionale: le immagini risiedono in regioni AWS specifiche, una caratteristica che riduce significativamente la latenza. Questo avviene perché le immagini non devono viaggiare attraverso la rete internet pubblica o tra data center AWS distanti quando vengono scaricate da servizi di calcolo, come ECS o EKS, che operano nella stessa regione.

Per gli sviluppatori che lavorano in ambienti locali, **LocalStack** estende la potenza di ECR al proprio ambiente di sviluppo. LocalStack permette di utilizzare le API di ECR per costruire e caricare immagini Docker in un registro locale, simulando il comportamento del servizio cloud e facilitando test e sviluppo offline.

Più in generale, un **Docker Registry** è un servizio per la gestione dei repository di immagini container. Le sue funzioni fondamentali includono la creazione di repository, il caricamento (_push_) e lo scaricamento (_pull_) di immagini, e la gestione degli accessi per controllare chi può interagire con esse. Esistono due tipologie principali di registry: quelli pubblici, come Docker Hub, che sono ospitati in cloud e non richiedono manutenzione da parte dell'utente, e quelli self-hosted. Questi ultimi offrono un controllo completo e possono essere personalizzati per esigenze specifiche. Ad esempio, un'azienda di servizi finanziari potrebbe scegliere un registro self-hosted per applicare controlli di accesso rigorosi e integrare uno scanner di vulnerabilità proprietario, garantendo la conformità con le policy di sicurezza interne—un livello di personalizzazione non disponibile sui registri pubblici come Docker Hub.

Dopo aver compreso il ruolo del registro, il passo successivo è definire il contenuto stesso di un'immagine container, un processo che viene formalizzato attraverso un file di istruzioni noto come Dockerfile.




# IL DOCKERFILE - DEFINIRE LE IMMAGINI CONTAINER
Il Dockerfile è lo strumento fondamentale per la creazione di immagini Docker riproducibili e standardizzate. Questo semplice file di testo agisce come una "ricetta" che definisce l'intero ambiente di esecuzione di un'applicazione containerizzata, specificando ogni passo necessario per assemblare l'immagine finale. L'uso di un Dockerfile garantisce che ogni build produca un risultato identico, eliminando le inconsistenze tra gli ambienti di sviluppo, test e produzione.

In dettaglio, un **Dockerfile** è un documento testuale che contiene una serie di direttive utilizzate dal motore Docker per costruire un'immagine. Queste istruzioni definiscono l'immagine di base da cui partire, i file da copiare, il software da installare, le variabili d'ambiente da impostare, le porte di rete da esporre e il comando da eseguire all'avvio del container.

Le istruzioni principali formano un linguaggio potente e flessibile per la costruzione delle immagini. L'istruzione `FROM <image>` è sempre la prima e specifica l'immagine di base su cui costruire la nuova. Per impostare la directory di lavoro predefinita all'interno dell'immagine, dove verranno eseguiti i comandi successivi, si utilizza `WORKDIR <path>`. Il comando `COPY <host-path> <image-path>` è fondamentale per copiare file e directory dal sistema host all'interno del filesystem dell'immagine. Infine, l'istruzione `RUN <command>` permette di eseguire comandi durante il processo di build, ed è tipicamente utilizzata per installare pacchetti software, dipendenze o eseguire script di configurazione.

Altre direttive gestiscono la configurazione dell'ambiente di esecuzione del container. Per definire variabili d'ambiente che saranno disponibili all'interno del container si usa `ENV <name> <value>`. La direttiva `EXPOSE <port-number>` documenta quali porte di rete il container esporrà, un'informazione utile sia per gli sviluppatori che per i sistemi di orchestrazione. Se è necessario eseguire comandi con un utente specifico, `USER <user-or-uid>` imposta l'utente predefinito per tutte le istruzioni successive. L'istruzione finale è solitamente `CMD ["<command>", "<arg1>"]`, che definisce il comando predefinito che verrà eseguito all'avvio di un container basato su questa immagine.

Ora che abbiamo esaminato la teoria dietro la definizione di un'immagine, è il momento di passare alla pratica per vedere come queste istruzioni vengono utilizzate per costruire e gestire un'immagine reale.




# FLUSSO OPERATIVO CON ECR: DALLA CREAZIONE AL PUSH DELL'IMMAGINE
Padroneggiare il ciclo di vita di un'immagine Docker è un'abilità strategica per qualsiasi sviluppatore che lavora con i container. Questo processo, che va dalla scrittura del codice di definizione al caricamento sicuro dell'immagine in un repository, costituisce il fondamento per il deployment di applicazioni containerizzate. Questo capitolo guiderà l'utente attraverso i passaggi pratici per costruire un'immagine, creare un repository ECR e infine caricarvi l'immagine.

Creazione dell'Immagine Docker

Il primo passo è creare un `Dockerfile` che definisca l'immagine. In questo esempio, creiamo un'immagine per una semplice applicazione web basata su `ubuntu:18.04`, che installa il server web Apache.

```
FROM public.ecr.aws/docker/library/ubuntu:18.04

# Install dependencies
RUN apt-get update && \
  apt-get -y install apache2

# Install apache and write hello world message
RUN echo 'Hello World!' > /var/www/html/index.html

# Configure apache
RUN echo '. /etc/apache2/envvars' > /root/run_apache.sh && \
  echo 'mkdir -p /var/run/apache2' >> /root/run_apache.sh && \
  echo 'mkdir -p /var/lock/apache2' >> /root/run_apache.sh && \
  echo '/usr/sbin/apache2 -D FOREGROUND' >> /root/run_apache.sh && \
  ln -sf /proc/self/fd/1 /var/log/apache2/access.log && \
  chmod 755 /root/run_apache.sh

EXPOSE 80
CMD /root/run_apache.sh
```

Questo Dockerfile è un eccellente esempio pratico. Il primo blocco `RUN` si occupa della gestione dei pacchetti: aggiorna l'indice dei pacchetti e installa `apache2`, preparando il software di base. Il secondo `RUN` è responsabile del deployment dell'applicazione: crea un semplice file `index.html` con il messaggio "Hello World!". Il terzo blocco `RUN` è più complesso e gestisce la configurazione: crea uno script di avvio (`run_apache.sh`) che imposta le variabili d'ambiente necessarie, crea le directory di runtime per Apache e avvia il server in primo piano. Infine, `EXPOSE 80` dichiara che il container esporrà la porta 80, e `CMD` imposta lo script di avvio come comando predefinito.

Per costruire l'immagine, si esegue il comando `docker build`:

```
$ docker build -t localstack-ecr-image .
```

Per verificare che l'immagine sia stata creata correttamente, si utilizza `docker images`. L'output mostrerà il nuovo repository `localstack-ecr-image` con il tag `latest`.

```
$ docker images

REPOSITORY                TAG       IMAGE ID       CREATED          SIZE
localstack-ecr-image      latest    38883941b8fa   1 minute ago     185MB
```

Creazione del Repository ECR

Successivamente, si crea un repository in ECR utilizzando l'API `CreateRepository` attraverso l'interfaccia a riga di comando. Il parametro `--image-scanning-configuration scanOnPush=true` abilita la scansione automatica delle vulnerabilità ogni volta che un'immagine viene caricata, un'importante misura di sicurezza.

```
awslocal ecr create-repository \
    --repository-name localstack-ecr-repository \
    --image-scanning-configuration scanOnPush=true
```

L'output di questo comando è un oggetto JSON che contiene i dettagli del repository. Il campo più importante è il `repositoryUri`, che rappresenta l'indirizzo univoco del repository. Questo URI è essenziale per poter caricare l'immagine.

```
{
    "repository": {
        "repositoryArn": "arn:aws:ecr:us-east-1:000000000000:repository/localstack-ecr-repository",
        "registryId": "000000000000",
        "repositoryName": "localstack-ecr-repository",
        "repositoryUri": "000000000000.dkr.ecr.us-east-1.localhost.localstack.cloud:4566/localstack-ecr-repository",
        "createdAt": "2023-07-24T16:58:36+05:30",
        "imageTagMutability": "MUTABLE",
        "imageScanningConfiguration": {
            "scanOnPush": true
        },
        "encryptionConfiguration": {
            "encryptionType": "AES256"
        }
    }
}
```

Caricamento dell'Immagine nel Repository

Per caricare un'immagine su un registro self-hosted o locale come quello di LocalStack, il nome dell'immagine deve seguire una specifica regola di nomenclatura: `[registry host]:[registry port]/[repository name]:[tag name]`.

È quindi necessario "ri-etichettare" (_retagging_) l'immagine locale con un nuovo alias che includa il `repositoryUri` ottenuto in precedenza. Si può pensare a questo processo non come alla copia dell'immagine, ma come all'aggiunta di una nuova etichetta di spedizione a un pacco esistente. Il pacco (i dati dell'immagine) rimane lo stesso, ma la nuova etichetta (`repositoryUri`) fornisce l'indirizzo corretto per inviarlo al registro ECR.

```
$ docker tag localstack-ecr-image 000000000000.dkr.ecr.us-east-1.localhost.localstack.cloud:4566/localstack-ecr-repository
```

Infine, si può caricare l'immagine nel repository ECR locale con il comando `docker push`. Per confermare il successo dell'operazione, si può elencare le immagini presenti nel repository tramite `awslocal ecr list-images`.

```
$ docker push 000000000000.dkr.ecr.us-east-1.localhost.localstack.cloud:4566/localstack-ecr-repository

$ awslocal ecr list-images --repository-name localstack-ecr-repository
```

In alternativa alla riga di comando, è possibile gestire i repository e le immagini anche tramite l'interfaccia web di LocalStack, nota come **Resource Browser**. Da qui è possibile eseguire azioni chiave come creare, visualizzare i dettagli ed eliminare un repository con pochi click.

Ora che l'immagine è archiviata e pronta per l'uso, il passo successivo è spostare l'attenzione dalla gestione delle singole immagini all'orchestrazione di container su larga scala, introducendo il servizio Amazon ECS.




# INTRODUZIONE AD AMAZON ELASTIC CONTAINER SERVICE (ECS)
Mentre un registro come ECR è essenziale per archiviare le immagini, per eseguire applicazioni containerizzate complesse in produzione è necessario un servizio di orchestrazione. Questi servizi gestiscono il deployment, l'esecuzione, la scalabilità e la disponibilità delle applicazioni, automatizzando compiti che altrimenti sarebbero manuali e soggetti a errori. **Amazon Elastic Container Service (ECS)** è la soluzione di AWS per l'orchestrazione di container.

Amazon ECS è un servizio di orchestrazione di container completamente gestito che permette di eseguire e gestire container Docker su un cluster. La sua caratteristica principale è che astrae la complessità della gestione dell'infrastruttura sottostante, fornendo tutti i servizi necessari per orchestrare i container in modo efficiente e affidabile.

Anche in questo caso, LocalStack offre un grande vantaggio per lo sviluppo locale, supportando le API di ECS. Ciò consente agli sviluppatori di creare e gestire cluster, definire task ed eseguire servizi direttamente sul proprio computer, replicando l'ambiente cloud e accelerando i cicli di sviluppo e test.

Le caratteristiche chiave di ECS ne definiscono il valore:

• **Orchestrazione e Flessibilità**: Semplifica l'orchestrazione dei container su un cluster di istanze EC2 o tramite la modalità serverless AWS Fargate, offrendo flessibilità nella scelta del modello operativo.

• **Scalabilità**: Permette lo scaling automatico delle applicazioni containerizzate per gestire picchi di traffico o variazioni del carico di lavoro in modo dinamico.

• **Bilanciamento del Carico**: Si integra nativamente con Elastic Load Balancing (ELB) per distribuire il traffico in entrata in modo uniforme tra i vari container, garantendo alta disponibilità e resilienza.

• **Ottimizzazione dei Costi**: Grazie a un modello di prezzo basato sull'uso effettivo delle risorse, si paga solo per ciò che si consuma, ottimizzando i costi operativi.

ECS offre due principali modalità di avvio (_Launch Types_) per l'esecuzione dei container: **EC2**, che offre pieno controllo sull'infrastruttura sottostante, e **Fargate**, una modalità serverless che elimina completamente la necessità di gestire server.

La scelta tra queste due modalità è cruciale e merita un'analisi più approfondita, poiché influisce direttamente sulla gestione operativa e sui costi.




# MODALITÀ DI AVVIO NI ECS: EC2 VS FARGATE
La scelta della modalità di avvio in ECS è una decisione strategica che influenza direttamente la gestione operativa, i costi e il livello di controllo sull'infrastruttura. Comprendere le differenze tra EC2 e Fargate è fondamentale per progettare un'architettura containerizzata efficiente e adatta alle proprie esigenze.

Modalità di Avvio EC2

Con la modalità di avvio EC2, si ha il pieno controllo sull'infrastruttura.

• **Gestione dell'Infrastruttura**: L'utente è responsabile del provisioning e della gestione di un cluster di istanze Amazon EC2 (macchine virtuali) che verranno unite al cluster ECS. Questo include la verifica della disponibilità di risorse sufficienti per eseguire i container.

• **Livello di Controllo**: Si ha un controllo granulare sulla configurazione, potendo scegliere i tipi di istanza (ad esempio, con GPU o custom AMI), applicare patch al sistema operativo, gestire gli aggiornamenti e la capacità di storage e networking.

• **Modello di Scaling**: Lo scaling è gestito configurando Auto Scaling Groups per le istanze EC2 o aggiustando manualmente la capacità del cluster.

• **Modello di Pricing**: Si paga per le istanze EC2 in esecuzione 24/7, indipendentemente dal fatto che siano pienamente utilizzate o inattive.

• **Casi d'uso ideali**: Questa modalità è adatta quando si necessita di un controllo fine sull'ambiente di calcolo, si hanno carichi di lavoro prevedibili che possono beneficiare di istanze riservate, o si desidera eseguire container su istanze EC2 già esistenti.

Modalità di Avvio AWS Fargate

AWS Fargate offre un'esperienza serverless, astraendo completamente l'infrastruttura.

• **Modello di Infrastruttura**: Non ci sono server da gestire. Si definisce semplicemente il task del container (specificando CPU, memoria e networking) e Fargate provvede a fornire le risorse di calcolo necessarie on-demand.

• **Livello di Controllo**: Il controllo è limitato. L'utente non ha accesso ai server sottostanti (niente SSH, niente patch del sistema operativo) e si concentra esclusivamente sulla gestione dell'immagine container e della task definition.

• **Modello di Scaling**: Lo scaling è automatico. I task vengono aggiunti o rimossi in base alle necessità senza doversi preoccupare delle dimensioni del cluster sottostante.

• **Modello di Pricing**: Si paga solo per le risorse di vCPU e memoria richieste mentre i task sono in esecuzione. Non ci sono costi per le risorse inattive.

• **Casi d'uso ideali**: Fargate è perfetto per applicazioni event-driven, carichi di lavoro con picchi imprevedibili, o quando si vuole minimizzare il tempo speso per la gestione dell'infrastruttura e concentrarsi unicamente sullo sviluppo dell'applicazione.

Il diagramma di confronto sintetizza visivamente queste differenze. Il modello **EC2** mostra le `Container Instance` (istanze EC2) che ospitano i task, gestite da un meccanismo di `Auto Scaling`. Il modello **Fargate**, invece, mostra direttamente una serie di task astratti, gestiti dal servizio senza che l'infrastruttura sottostante sia visibile o accessibile.

Dopo aver chiarito come i container vengono eseguiti, è essenziale comprendere i componenti logici che ECS utilizza per organizzarli, come i cluster e le task definition




# I COMPONENTI FONDAMENTALI DI ECS
Per orchestrare e gestire le applicazioni containerizzate in modo efficace, Amazon ECS si basa su una serie di componenti logici interconnessi. La comprensione di cluster, task definition, servizi e container instances è essenziale per sfruttare appieno la potenza del servizio.

Il più alto livello di organizzazione in ECS è il **Cluster**, un raggruppamento logico di risorse e task. Il suo ruolo principale è quello di fornire un meccanismo di separazione per isolare ambienti, gestire le risorse e organizzare i container. Ad esempio, è prassi comune creare cluster distinti per gli ambienti di sviluppo e produzione di una web application. Questo isolamento garantisce che le modifiche apportate nell'ambiente di sviluppo non possano accidentalmente influenzare quello di produzione.

Per eseguire un'applicazione all'interno di un cluster, è prima necessario descriverla utilizzando una **Task Definition**. Questa funge da "blueprint" per l'applicazione, definendo come uno o più container dovrebbero comportarsi. Si tratta di un documento, tipicamente in formato JSON, che specifica parametri cruciali come le immagini container da utilizzare, i limiti di CPU e memoria, le variabili d'ambiente da iniettare e le configurazioni di rete. Una singola Task Definition può descrivere un'applicazione composta da più container che devono collaborare.

Una volta definita l'applicazione, entra in gioco il **Servizio ECS**. Un servizio ha il compito di mantenere in esecuzione un numero desiderato di copie di un task, basandosi su una specifica Task Definition. Se un task si arresta inaspettatamente, il servizio lo riavvia automaticamente per mantenere lo stato desiderato. Inoltre, i servizi gestiscono lo scaling automatico, aggiungendo o rimuovendo task in base a metriche predefinite (come l'utilizzo della CPU) per adattarsi dinamicamente alle variazioni del carico di lavoro.

Infine, le **Container Instances** sono gli host su cui vengono effettivamente eseguiti i container. A seconda della modalità di avvio scelta, possono essere istanze EC2 gestite dall'utente o task Fargate gestiti da AWS. Nel caso della modalità EC2, su ogni istanza viene eseguito un processo specializzato chiamato **ECS Agent**. Questo agente è responsabile della comunicazione con il control plane di ECS, della gestione del ciclo di vita dei task sull'istanza (avvio e arresto dei container) e del reporting dello stato.

Ora che abbiamo definito questi componenti astratti, vediamo come vengono creati e orchestrati nella pratica attraverso comandi specifici.




# FLUSSO OPERATIVO CON ECS: CREAZIONE E GESTIONE DI UN SERVIZIO
Dopo aver esplorato i concetti teorici di ECR ed ECS, questo capitolo finale rappresenta la sintesi pratica, mettendo insieme tutti gli elementi. Seguiremo i passaggi per creare un cluster, definire e registrare un task, lanciare un servizio e infine monitorarlo e aggiornarlo.

1. Creazione del Cluster

Il primo passo è creare un cluster, che funge da raggruppamento logico per i nostri servizi e task. Il seguente comando crea un cluster vuoto chiamato `mycluster`.

```
$ awslocal ecs create-cluster --cluster-name mycluster
```

2. Definizione e Registrazione del Task

I container vengono definiti all'interno di un file JSON di **task definition**. Questo file specifica tutti i parametri necessari per l'esecuzione.

```
{
    "family": "myfamily",
    "containerDefinitions": [
        {
            "name": "server",
            "image": "000000000000.dkr.ecr.us-east-1.localhost.localstack.cloud:4566/localstack-ecr-repository:latest",
            "cpu": 10,
            "memory": 10,
            "portMappings": [
                {
                    "containerPort": 80,
                    "protocol": "tcp",
                    "appProtocol": "http"
                }
            ],
            "essential": true,
            "logConfiguration": {
                "logDriver": "awslogs",
                "options": {
                    "awslogs-create-group": "true",
                    "awslogs-group": "myloggroup",
                    "awslogs-stream-prefix": "myprefix",
                    "awslogs-region": "us-east-1"
                }
            }
        }
    ],
    "requiresCompatibilities": ["FARGATE"]
}
```

I parametri principali sono:

• `family`: Un nome che raggruppa diverse versioni della stessa task definition.

• `containerDefinitions`: Un elenco di definizioni di container. Qui specifichiamo il `name` del container, l'`image` da usare (l'URI del nostro repository ECR), i limiti di `cpu` e `memory`, le `portMappings` per esporre la porta 80 e la `logConfiguration` per inviare i log a CloudWatch.

• `requiresCompatibilities`: Con `["FARGATE"]`, indichiamo a ECS che questo task è progettato per essere eseguito con la modalità di avvio Fargate.

Per registrare questa definizione, si usa il comando:

```
$ awslocal ecs register-task-definition --cli-input-json file://task_definition.json
```

È importante notare che le task definition sono **immutabili**. Ogni volta che si registra una nuova versione con la stessa `family`, viene creato un nuovo numero di revisione. Inoltre, la configurazione dei log specificata creerà automaticamente un log group in CloudWatch per monitorare l'output del container.

3. Avvio del Servizio

Con il cluster e la task definition pronti, possiamo avviare un servizio per eseguire e mantenere i nostri container.

```
$ awslocal ecs create-service --service-name myservice --cluster mycluster \
  --task-definition myfamily --launch-type FARGATE --desired-count 1 \
  --network-configuration "awsvpcConfiguration={subnets=[\"<SUBNET-ID>\"],securityGroups=[\"SG-ID\"],assignPublicIp=\"ENABLED\"}"
```

Questo comando crea un servizio chiamato `myservice` nel cluster `mycluster`, utilizzando la task definition `myfamily`. Specifichiamo `launch-type FARGATE`, un `desired-count` di 1 (per avviare una singola istanza del task) e la `network-configuration`. Quest'ultima è obbligatoria per Fargate perché i task vengono eseguiti in un ambiente gestito da AWS e necessitano di questa configurazione per essere collegati a una VPC, ricevere un'interfaccia di rete (ENI) e ottenere indirizzi IP e regole di sicurezza che li rendano accessibili.

4. Monitoraggio dei Log

Per visualizzare i log generati dal container, possiamo interrogare il log group di CloudWatch specificato nella task definition.

```
$ awslocal logs filter-log-events --log-group-name myloggroup
```

È anche possibile limitare l'output o estrarre solo il campo del messaggio per una lettura più pulita.

```
# Limita a 20 linee
$ awslocal logs filter-log-events --log-group-name myloggroup | select -first 20

# Estrae solo il messaggio
$ awslocal logs filter-log-events --log-group-name myloggroup --query "events[].message"
```

Nota: il comando `select -first 20` utilizza una pipeline di PowerShell per limitare l'output; su altri terminali (come bash), si potrebbe usare `| head -n 20`. L'opzione `--query` utilizza JMESPath, un potente linguaggio di filtraggio integrato nella AWS CLI, per estrarre campi specifici dalla risposta JSON.

5. Aggiornamento del Servizio

Infine, ECS consente di aggiornare la configurazione di un servizio in esecuzione. Ad esempio, per scalare orizzontalmente l'applicazione aumentando il numero di task da 1 a 2, si può modificare il `desired-count` con il seguente comando:

```
$ awslocal ecs update-service --service myservice --cluster mycluster --desired-count 2
```

ECS si occuperà di avviare un nuovo task per raggiungere lo stato desiderato di due istanze in esecuzione.
