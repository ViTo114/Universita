# INTRODUZIONE AD APACHE HIVE
Apache Hive si posiziona come uno strumento strategico all'interno dell'ecosistema Hadoop, progettato per semplificare l'analisi di grandi moli di dati. La sua funzione principale è quella di fornire un livello di astrazione sopra HDFS (Hadoop Distributed File System), nascondendo la complessità intrinseca della programmazione MapReduce. Attraverso un'interfaccia dichiarativa basata su SQL, nota come HiveQL, Hive rende l'interrogazione e l'analisi dei Big Data accessibile a un pubblico molto più vasto, che include analisti di dati e analisti di business, i quali possono sfruttare le loro competenze SQL esistenti senza dover scrivere codice a basso livello. Questa astrazione è di importanza strategica poiché democratizza l'accesso ai dati, accelerando i cicli di analisi e la generazione di valore.

Ruolo e Funzionalità Principali

Hive agisce come un _data warehouse layer_ costruito direttamente su HDFS. Invece di memorizzare i dati in un proprio formato proprietario, Hive impone uno schema logico (composto da tabelle, colonne e partizioni) su file già presenti in HDFS. Il suo ruolo non è quello di un database tradizionale, ma di un motore di interrogazione che traduce le query scritte in HiveQL in processi di esecuzione distribuiti. A seconda della configurazione, questi processi possono essere job MapReduce, task Tez o applicazioni Spark, orchestrati da YARN per l'allocazione delle risorse.

Le sue funzionalità chiave includono:

• **Data Modeling**: Permette di definire strutture logiche come tabelle, che vengono mappate su directory in HDFS, e partizioni, che corrispondono a sottodirectory, ottimizzando le performance delle query.

• **Interfaccia SQL-like**: HiveQL semplifica l'interrogazione dei Big Data, rendendola intuitiva per chi ha familiarità con il linguaggio SQL.

• **Estensibilità**: Supporta l'integrazione di logica personalizzata attraverso **UDF** (User-Defined Functions), che operano su una singola riga; **UDAF** (User-Defined Aggregation Functions), che aggregano valori su più righe; e **UDTF** (User-Defined Table-Generating Functions), che trasformano una singola riga in input in multiple righe in output.

• **Gestione di Tabelle Esterne**: Hive può lavorare con "tabelle esterne", ovvero può applicare uno schema a dati già esistenti in HDFS senza prenderne possesso, una caratteristica che lo rende estremamente versatile per l'analisi di dati provenienti da fonti diverse.

Casi d'Uso Tipici

Grazie alla sua architettura orientata al batch, Hive eccelle in scenari dove la latenza non è il fattore critico, ma lo è la capacità di processare enormi volumi di dati. I casi d'uso più comuni includono:

• **Analisi Batch**: Esecuzione di query complesse su terabyte o petabyte di dati storici per estrarre insight.

• **Processi ETL (Extract, Transform, Load)**: Utilizzato come motore per pipeline di trasformazione dati, dove i dati grezzi vengono puliti, arricchiti e strutturati per l'analisi successiva.

• **Reportistica Pianificata**: Generazione di report periodici (giornalieri, settimanali, mensili) che richiedono l'aggregazione di grandi dataset.

• **Query Ad-Hoc**: Permette ad analisti di esplorare i dati in modo interattivo (con la consapevolezza di una latenza non real-time) per rispondere a specifiche domande di business.

Per sfruttare appieno la potenza di Hive, è essenziale capire come esso organizza fisicamente i dati. Ciò introduce la necessità di approfondire i concetti di tabelle, partizioni e bucket, che sono fondamentali per l'ottimizzazione delle query.




# IL MODELLO DATI E L'ARCHITETTURA DI HIVE
Per utilizzare Hive in modo efficace, non è sufficiente conoscerne la sintassi SQL; è fondamentale comprendere come le sue strutture logiche (come le tabelle) vengano mappate su strutture fisiche (directory e file in HDFS) e quali siano i componenti architetturali che ne orchestrano il funzionamento. Questa conoscenza è la chiave per scrivere query performanti e per ottimizzare l'organizzazione dei dati, garantendo che il sistema possa elaborare le richieste nel modo più efficiente possibile.

Il Modello Dati di Hive: Tabelle, Partizioni e Bucket

Il modello dati di Hive è progettato per ottimizzare l'accesso a grandi volumi di dati su un file system distribuito.

• **Tabelle**: Una tabella in Hive non è altro che un'astrazione logica che corrisponde a una **directory in HDFS**. Tutti i file contenuti in quella directory sono considerati parte della tabella. Hive non sposta né gestisce i dati; si limita a memorizzare lo schema e i metadati che descrivono come interpretare i file.

• **Partizioni**: Il partizionamento è la tecnica di ottimizzazione più importante in Hive. Le colonne di partizione definite nello schema di una tabella si materializzano come **sottodirectory** all'interno della directory principale della tabella. Ad esempio, una tabella partizionata per data (`ds`) avrà sottodirectory come `/path/to/table/ds=2020-03-10`, `/path/to/table/ds=2020-03-11`, e così via. Quando una query include un filtro sulla colonna di partizione (es. `WHERE ds = '2020-03-10'`), Hive può saltare la scansione di tutte le altre directory, un processo noto come _directory pruning_, che migliora drasticamente le prestazioni riducendo l'I/O. Questa capacità rende il partizionamento una scelta strategica fondamentale per la gestione efficiente dei dati storici.

• **Bucket**: Il bucketing è un'ulteriore suddivisione dei dati all'interno di una partizione. I dati vengono distribuiti in un numero fisso di file (i _bucket_) applicando una **funzione di hash** su una o più colonne al momento del caricamento. Questo significa che ogni record con lo stesso valore nella colonna di bucketing finirà sempre nello stesso file. Questa organizzazione deterministica ottimizza ulteriormente le query, specialmente per le operazioni di join su tabelle di grandi dimensioni, rendendolo uno strumento strategico per ottimizzare i flussi di lavoro di aggregazione su larga scala.

Componenti Chiave dell'Architettura di Hive

L'architettura di Hive è composta da diversi componenti che collaborano per eseguire una query:

• **Clienti**: Sono le interfacce attraverso cui gli utenti interagiscono con Hive. Includono la **CLI (Command-Line Interface)** per l'esecuzione interattiva o in batch di script, il **Thrift Server** che abilita la comunicazione cross-linguaggio, e i **driver JDBC/ODBC** per l'integrazione con applicazioni esterne.

• **Driver**: È il cuore del sistema. Riceve la query dal client e ne gestisce l'intero ciclo di vita. Contiene:

    ◦ **Compilatore**: Esegue il parsing della query HiveQL, la trasforma in un albero di operatori e ne valida la semantica.

    ◦ **Ottimizzatore**: Applica una serie di ottimizzazioni logiche (indipendenti dal motore di esecuzione, come il **predicate pushdown**) e fisiche (specifiche per il motore scelto) per generare il piano di esecuzione più efficiente.

    ◦ **Motore di Esecuzione**: Traduce il piano ottimizzato in un flusso di lavoro eseguibile (ad es. job MapReduce) e lo sottomette al cluster.

• **Metastore**: È il componente che memorizza tutti i metadati.

• **Infrastruttura Hadoop**: Hive si appoggia a **YARN** per l'allocazione delle risorse del cluster e a **HDFS** per la lettura e la scrittura dei dati.

Il Ruolo Cruciale del Metastore

Il **Metastore** è il cervello di Hive. Memorizza in un database relazionale (come MySQL o PostgreSQL) tutte le informazioni necessarie per interpretare i dati, tra cui gli schemi delle tabelle, le definizioni delle partizioni, i formati dei file e la loro posizione in HDFS. Senza di esso, nessuna query potrebbe essere eseguita. Quando una query viene sottomessa, il Driver consulta il Metastore per sapere **quali file deve scansionare e, aspetto cruciale, come deserializzare i record contenuti in essi**. Può essere configurato in modalità _embedded_ (legata a una singola istanza Hive, adatta solo per test) o _remota_ (la configurazione raccomandata per ambienti di produzione), che garantisce scalabilità e accesso centralizzato ai metadati.

Comprendere come i dati sono strutturati logicamente e fisicamente ci porta a un'altra scelta fondamentale per le prestazioni: il formato di file con cui vengono effettivamente memorizzati.




# FORMATI DI ARCHIVIAZIONE E FLUSSO DI ESECUZIONE DELLE QUERY
Oltre alla strutturazione dei dati tramite partizioni e bucket, un altro fattore che ha un impatto significativo sulle prestazioni di Hive è la scelta del formato di archiviazione dei file. Allo stesso tempo, comprendere il flusso di esecuzione di una query permette di demistificare come un semplice comando SQL venga trasformato in una complessa operazione di calcolo distribuito su un cluster Hadoop.

Formati di Archiviazione: Row-Based vs Columnar

Hive supporta diversi formati di file, che possono essere raggruppati in due categorie principali: basati su righe (Row-Based) e colonnari (Columnar).

Formati Row-Based

In questi formati, i dati di ogni riga vengono memorizzati in modo contiguo. Sono ideali per carichi di lavoro in cui è necessario accedere a tutte le colonne di una riga.

• **TEXTFILE**: Il formato di default. È testo semplice, leggibile dall'uomo, ma inefficiente per l'elaborazione su larga scala a causa della mancanza di compressione e di metadati. Utile per test di base.

• **SEQUENCEFILE**: Un formato binario basato su righe che supporta la compressione. È _splittable_, il che lo rende adatto per l'elaborazione parallela in MapReduce. Viene spesso utilizzato per lo storage intermedio durante i workflow ETL.

• **AVRO**: Formato che include lo schema dei dati nel file stesso, supportando l'evoluzione dello schema nel tempo. È un formato bilanciato per operazioni di lettura e scrittura, ideale per lo scambio di dati tra sistemi diversi.

Formati Columnar

In questi formati, i valori di ogni colonna vengono memorizzati in modo contiguo. Questo approccio offre enormi vantaggi per le query analitiche, che tipicamente accedono solo a un sottoinsieme delle colonne.

• **ORC (Optimized Row Columnar)**: Offre un'elevata compressione e prestazioni di lettura molto veloci grazie a indici interni e al _predicate pushdown_ (la capacità di filtrare i dati a livello di storage). È la scelta ideale per carichi di lavoro analitici read-heavy.

• **PARQUET**: Un altro formato colonnare, efficiente nella gestione di strutture dati annidate e ampiamente supportato nell'ecosistema di analisi moderno (incluso Spark). È ottimo per l'elaborazione analitica con dati complessi.

**Regola Generale**: Scegliere un formato **Row-Based** per carichi di lavoro con scritture frequenti o che richiedono l'accesso all'intera riga. Scegliere un formato **Columnar** per carichi di lavoro analitici su larga scala, dove la compressione è importante e le query accedono a un sottoinsieme selettivo di colonne, rendendolo la scelta standard per i data warehouse moderni. I vantaggi dei formati colonnari sono pienamente realizzati quando combinati con le ottimizzazioni delle query di Hive, come il _projection pruning_, che evita la lettura di colonne non necessarie, e il _predicate pushdown_, che filtra i dati alla fonte, riducendo drasticamente l'I/O.

Flusso di Elaborazione di una Query Hive

Il ciclo di vita di una query HiveQL si articola in tre fasi principali:

1. **Compilazione**: Quando la query viene sottomessa, il Driver di Hive la analizza sintatticamente e semanticamente. La query viene validata rispetto ai metadati presenti nel Metastore e trasformata in un **piano logico**, ovvero una rappresentazione ad albero degli operatori logici necessari per soddisfare la richiesta (es. scan, filter, join).

2. **Ottimizzazione**: Il piano logico viene processato dall'Ottimizzatore, che applica una serie di regole per renderlo più efficiente. Le **ottimizzazioni logiche** sono indipendenti dal motore di esecuzione e includono tecniche come la potatura delle proiezioni (rimozione di colonne non necessarie) e la riscrittura dei join. Successivamente, vengono applicate le **ottimizzazioni fisiche**, che sono specifiche per il motore di esecuzione scelto (es. MapReduce) e definiscono come implementare concretamente gli operatori logici.

3. **Esecuzione**: Il piano ottimizzato viene tradotto in una serie di **stage** (o task eseguibili). Questi stage vengono sottomessi al motore di esecuzione (es. un job MapReduce), che si coordina con **YARN** per l'allocazione delle risorse (container) nel cluster. I dati vengono letti da **HDFS**, elaborati dai task, e i risultati intermedi vengono scritti nuovamente su HDFS. Infine, il risultato finale viene restituito al client o persistito in una nuova tabella.

Sebbene Hive sia uno strumento potente e consolidato per l'analisi batch, l'ecosistema Big Data si è evoluto. Esistono framework più moderni e veloci, progettati per superare i limiti di MapReduce, e tra questi il più importante è Apache Spark.





# INTRODUZIONE A APACHE SPARK
Apache Spark si è affermato come il framework di calcolo distribuito di riferimento per l'elaborazione dei Big Data. Sviluppato presso l'UC Berkeley nel 2009 e diventato parte della Apache Foundation nel 2010, è nato per superare i limiti di latenza e rigidità del paradigma MapReduce di Hadoop, introducendo un modello di calcolo più flessibile e performante. La sua importanza strategica risiede principalmente nella sua **velocità**, ottenuta grazie a un uso intelligente dell'elaborazione in memoria, e nella sua **versatilità**, che gli consente di supportare una vasta gamma di carichi di lavoro (batch, interattivi, streaming, machine learning) attraverso un set di API unificate.

Architettura Master-Slave di Spark

L'architettura di Spark si basa su un modello master-slave, orchestrato da tre componenti principali che collaborano per eseguire un'applicazione distribuita.

1. **Programma Driver**: È il processo principale che gestisce l'esecuzione dell'applicazione Spark. Il driver è responsabile del parsing del codice dell'utente, della creazione di un piano di esecuzione e della sua suddivisione in task distribuiti. Coordina l'esecuzione di questi task sui nodi del cluster e raccoglie i risultati.

2. **Executor**: Sono processi di lavoro che vengono eseguiti sui nodi del cluster (worker nodes). Ogni executor dispone di una certa quantità di CPU e memoria e ha il compito di eseguire i task assegnatigli dal driver. Gli executor comunicano lo stato dei task e restituiscono i risultati al programma driver.

3. **Cluster Manager**: È il componente responsabile dell'allocazione delle risorse fisiche del cluster. Spark può interfacciarsi con diversi cluster manager, come **YARN** (il gestore di risorse di Hadoop), **Apache Mesos**, o il suo **Standalone Scheduler** integrato. Il driver richiede risorse al cluster manager, che a sua volta le alloca per avviare gli executor sui nodi di lavoro.

Modalità di Esecuzione e API Supportate

Un'applicazione Spark può essere eseguita in diverse modalità, a seconda delle esigenze:

• **Local Mode**: Sia il driver che gli executor vengono eseguiti all'interno di un'unica JVM sulla macchina locale. È ideale per lo sviluppo, il testing e il debugging rapidi, in quanto non richiede un'infrastruttura cluster.

• **Client Mode**: Il driver viene eseguito sulla macchina locale (il client da cui si lancia l'applicazione), mentre gli executor vengono eseguiti sui nodi del cluster. È utile per scenari interattivi dove il driver ha bisogno di accedere a dati o configurazioni locali.

• **Cluster Mode**: Sia il driver che gli executor vengono eseguiti all'interno del cluster. Questa è la modalità più comune per gli ambienti di produzione, poiché garantisce che l'intera applicazione sia gestita dal cluster manager, rendendola resiliente a eventuali problemi di rete tra il client e il cluster.

Spark offre API in diversi linguaggi di programmazione, tra cui **Scala** (il linguaggio primario), **Java**, **Python** e **R**, permettendo a sviluppatori con background diversi di utilizzare il framework. Fornisce inoltre delle shell interattive (`spark-shell` per Scala, `pyspark` per Python) per la prototipazione rapida e l'analisi esplorativa dei dati.

SparkContext e SparkSession

Per interagire con il motore di Spark, un'applicazione deve prima connettersi al cluster.

• **SparkContext**: Storicamente, lo `SparkContext` era l'interfaccia principale e il punto di ingresso per qualsiasi applicazione Spark. È l'oggetto, creato dal programma driver, responsabile della connessione al cluster manager e della coordinazione delle risorse.

• **SparkSession**: A partire dalla versione 2.x, lo `SparkSession` è diventato il punto di ingresso unificato. Questa nuova astrazione ingloba al suo interno lo `SparkContext` e fornisce un'interfaccia unica per interagire con tutte le funzionalità di Spark (inclusi Spark SQL, Streaming, ecc.), semplificando notevolmente la configurazione e lo sviluppo delle applicazioni.

Il cuore del modello di programmazione di Spark e la chiave della sua efficienza risiedono in un'astrazione fondamentale: il Resilient Distributed Dataset (RDD), che sarà l'argomento centrale del prossimo capitolo.




# I RESILIENT DISTRIBUTED DATASET (RDD)
Al centro del modello di programmazione di Apache Spark si trova il **Resilient Distributed Dataset (RDD)**, l'astrazione dati fondamentale del framework. Un RDD rappresenta una collezione di oggetti, partizionata e immutabile, che può essere elaborata in parallelo sui nodi di un cluster. Comprendere le proprietà degli RDD è essenziale per scrivere applicazioni Spark che siano non solo efficienti, ma anche robuste e tolleranti ai guasti.

Proprietà Fondamentali di un RDD

Il nome stesso di RDD ne descrive le caratteristiche chiave:

• **Resiliente (Resilient)**: Gli RDD sono tolleranti ai guasti. La loro resilienza non è ottenuta attraverso la replica dei dati (come in HDFS), ma attraverso il concetto di _lineage_ (lignaggio). Spark tiene traccia della sequenza di trasformazioni che hanno generato ogni RDD, permettendogli di ricalcolare una partizione persa in caso di guasto di un nodo.

• **Distribuito (Distributed)**: I dati all'interno di un RDD sono suddivisi in partizioni logiche. Ciascuna partizione può risiedere e essere elaborata su un nodo diverso del cluster, consentendo un'enorme parallelizzazione del calcolo.

• **Immutabile (Immutable)**: Un RDD, una volta creato, non può essere modificato. Qualsiasi operazione di trasformazione applicata a un RDD non cambia l'RDD originale, ma crea sempre un nuovo RDD. Questa proprietà garantisce la coerenza e semplifica la gestione dei guasti.

Il Modello di Programmazione: Trasformazioni e Azioni

Il modello di programmazione degli RDD si basa su due tipi di operazioni:

1. **Trasformazioni (Transformations)**: Sono operazioni che, applicate a un RDD, ne creano uno nuovo. Esempi comuni sono `map()`, `filter()` e `groupByKey()`. Le trasformazioni sono _pigre_ (_lazy_), il che significa che Spark non esegue immediatamente il calcolo. Invece, registra la trasformazione nel _lineage_ e costruisce un grafo di esecuzione (DAG - Directed Acyclic Graph).

2. **Azioni (Actions)**: Sono operazioni che innescano l'effettiva esecuzione del calcolo descritto dal DAG e restituiscono un risultato al programma driver o scrivono i dati su un sistema di storage esterno. Esempi sono `collect()`, `count()` e `saveAsTextFile()`.

Questa separazione tra la definizione del calcolo (trasformazioni) e la sua esecuzione (azioni) è una delle caratteristiche più potenti di Spark. Permette al suo ottimizzatore di analizzare l'intero flusso di lavoro prima di eseguirlo, riordinando o combinando le operazioni per massimizzare l'efficienza.

Modello di Esecuzione e Recupero dai Guasti

Quando un'azione viene invocata, Spark analizza il DAG delle trasformazioni e lo suddivide in _stage_. Ogni stage contiene un gruppo di task che possono essere eseguiti in parallelo senza richiedere uno scambio di dati tra i nodi (shuffle). Ogni task elabora una singola partizione dell'RDD.

La resilienza è garantita dal **lineage**. Ogni RDD mantiene metadati specifici che descrivono come è stato derivato, includendo **riferimenti agli RDD genitori, la logica di calcolo applicata, le informazioni sul partizionamento e le preferenze di località dei dati (preferred locations)**. Se un nodo del cluster fallisce e una partizione viene persa, Spark non ha bisogno di ricaricare i dati dall'origine o di fare affidamento su repliche. Può semplicemente usare il lineage per **ricalcolare la partizione persa** a partire dai dati ancora disponibili, garantendo la tolleranza ai guasti in modo efficiente e senza la necessità di replicare l'intero dataset.

Il prossimo capitolo esplorerà in dettaglio i diversi tipi di trasformazioni e azioni disponibili per manipolare gli RDD, fornendo una panoramica completa delle operazioni che costituiscono la base della programmazione in Spark.




# TRASFORMAZIONI E AZIONI SUGLI RDD
La programmazione in Spark ruota attorno alla distinzione fondamentale tra trasformazioni, che definiscono un nuovo RDD in modo pigro, e azioni, che attivano l'esecuzione del calcolo. Comprendere la differenza tra i vari tipi di operazioni e il loro impatto sulle performance è cruciale per sviluppare applicazioni Spark efficienti. Le trasformazioni, in particolare, si dividono in due categorie principali: strette (narrow) e ampie (wide).

Trasformazioni Strette (Narrow Transformations)

Le trasformazioni strette, o _narrow_, sono operazioni in cui **ogni partizione dell'RDD di output dipende al massimo da una partizione dell'RDD di input**. Questo significa che il calcolo può avvenire in modo indipendente su ciascuna partizione, senza la necessità di scambiare dati tra i diversi nodi del cluster. Poiché non richiedono comunicazione di rete, le trasformazioni strette sono molto efficienti e possono essere eseguite in pipeline all'interno di un unico _stage_ di esecuzione, rendendole ideali per operazioni di filtro e mappatura a basso costo.

Esempi comuni di trasformazioni strette includono:

• `map()`: Applica una funzione a ogni elemento dell'RDD, restituendo un nuovo RDD con lo stesso numero di elementi.

• `flatMap()`: Simile a `map()`, ma la funzione applicata può restituire zero o più elementi per ogni elemento di input.

• `filter()`: Restituisce un nuovo RDD contenente solo gli elementi che soddisfano una data condizione.

• `union()`: Combina due RDD in uno solo, mantenendo tutti gli elementi di entrambi (inclusi i duplicati).

Trasformazioni Ampie (Wide Transformations)

Le trasformazioni ampie, o _wide_, sono operazioni in cui **una partizione dell'RDD di output può dipendere da più partizioni dell'RDD di input**. Per calcolare il risultato, Spark deve raggruppare i dati con la stessa chiave, che potrebbero trovarsi su nodi diversi. Questo richiede un **rimescolamento dei dati (shuffle)** attraverso la rete, un'operazione che consiste nel ridistribuire i dati tra gli executor. Lo shuffle è costoso in termini di I/O di rete e disco e, per questo motivo, le trasformazioni ampie segnano il confine tra gli _stage_ nel piano di esecuzione di Spark.

Esempi comuni di trasformazioni ampie includono:

• `distinct()`: Rimuove gli elementi duplicati da un RDD.

• `sortBy()`: Ordina gli elementi dell'RDD.

• `intersection()`: Restituisce un nuovo RDD contenente solo gli elementi comuni a due RDD.

• `subtract()`: Rimuove da un RDD gli elementi contenuti in un altro.

• Operazioni su RDD chiave-valore come `groupByKey()` e `reduceByKey()`.

Le Principali Azioni

Le azioni sono le operazioni che, una volta invocate, scatenano l'esecuzione dell'intero grafo di trasformazioni (DAG) e producono un risultato. Le azioni più comuni sono:

• `collect()`: Restituisce tutti gli elementi dell'RDD al programma driver come una collezione. Da usare con cautela, poiché può causare un out-of-memory nel driver se l'RDD è molto grande.

• `count()`: Restituisce il numero totale di elementi nell'RDD.

• `take(n)`: Restituisce i primi `n` elementi dell'RDD.

• `first()`: Restituisce il primo elemento dell'RDD (equivalente a `take(1)`).

• `saveAsTextFile(path)`: Scrive gli elementi dell'RDD in file di testo in una directory specificata.

• `reduce(func)`: Aggrega gli elementi dell'RDD utilizzando una funzione associativa e commutativa, restituendo il risultato finale al driver.

Mentre le trasformazioni viste finora operano su RDD generici, molte analisi sui dati richiedono operazioni su dati strutturati come coppie chiave-valore. Per questi scenari, Spark offre un tipo speciale di RDD, i Pair RDD, che sono ottimizzati per questo tipo di operazioni e verranno discussi nel prossimo capitolo.



# GLI RDD DI COPPIE (PAIR RDD) E OTTIMIZZAZONE DELLE PRESTAZIONI
Molte operazioni di analisi dei dati, come aggregazioni, raggruppamenti e join, si basano su dati strutturati in coppie chiave-valore. Per gestire questi scenari in modo efficiente, Spark fornisce un'astrazione specializzata chiamata **Pair RDD**. In questo capitolo esploreremo le operazioni ottimizzate disponibili per i Pair RDD e altre tecniche generali, come il caching e il checkpointing, per migliorare le prestazioni delle applicazioni Spark.

Cosa Sono i Pair RDD e Come si Creano

Un Pair RDD è semplicemente un RDD i cui elementi sono **tuple chiave-valore** (tecnicamente, tuple di due elementi). Questa struttura permette a Spark di eseguire trasformazioni ottimizzate che agiscono specificamente sulla chiave di ogni elemento. Ad esempio, quando si raggruppano o si uniscono dati, Spark sa che deve spostare attraverso la rete solo i valori associati alla stessa chiave, rendendo le operazioni di _shuffle_ più efficienti. Un Pair RDD si crea tipicamente da una collezione di tuple o trasformando un RDD esistente con un'operazione come `map()` che restituisce coppie chiave-valore.

Confronto tra `groupByKey()` e `reduceByKey()`

Due delle trasformazioni _wide_ più importanti per i Pair RDD sono `groupByKey()` e `reduceByKey()`. Sebbene possano sembrare simili, il loro impatto sulle prestazioni è drasticamente diverso.

• `groupByKey()`: Questa operazione raggruppa tutti i valori associati a una data chiave. Per fare ciò, Spark deve spostare tutti i valori per ogni chiave attraverso la rete verso un singolo nodo, dove vengono raccolti in un iteratore. Questo può causare un notevole traffico di rete e potenziali problemi di memoria se ci sono molteplici valori per una singola chiave.

• `reduceByKey()`: Questa operazione esegue un'aggregazione dei valori per ogni chiave utilizzando una funzione associativa. La sua grande efficienza deriva dal fatto che sfrutta il pattern "combiner" di MapReduce, eseguendo un'**aggregazione locale** su ciascuna partizione _prima_ di inviare i risultati intermedi attraverso la rete. In questo modo, la quantità di dati da trasferire durante lo shuffle viene drasticamente ridotta. Per questo motivo, **reduceByKey()** **è quasi sempre preferibile a** **groupByKey()** quando l'obiettivo è aggregare i dati, rendendola la scelta standard per ottimizzare i flussi di lavoro di aggregazione su larga scala.

Altre Trasformazioni su Pair RDD

Oltre a `reduceByKey()`, i Pair RDD supportano altre operazioni ottimizzate, tra cui:

• `sortByKey()`: Ordina l'RDD in base alla chiave, in modo ascendente o discendente.

• `join()`: Combina due Pair RDD in base alle chiavi comuni, creando un nuovo RDD con elementi del tipo `(chiave, (valore1, valore2))`.

Tecniche di Caching e Checkpointing

Per ottimizzare ulteriormente le prestazioni, specialmente quando un RDD viene utilizzato più volte in un'applicazione, Spark offre due meccanismi principali:

• **Caching (o Persistenza)**: Questa tecnica consente di **memorizzare un RDD in memoria o su disco** la prima volta che viene calcolato. Quando un'azione successiva richiede lo stesso RDD, Spark può riutilizzare la versione salvata invece di ricalcolarlo dall'inizio. Esistono diversi livelli di persistenza che offrono un compromesso tra uso della memoria, CPU e velocità di accesso:

    ◦ `MEMORY_ONLY`: Memorizza gli oggetti RDD deserializzati nella memoria della JVM.

    ◦ `MEMORY_ONLY_SER`: Memorizza gli oggetti serializzati in memoria (meno ingombrante ma più costoso per la CPU).

    ◦ `MEMORY_AND_DISK`: Memorizza le partizioni in memoria; se non c'è abbastanza spazio, quelle in eccesso vengono salvate su disco.

    ◦ `MEMORY_AND_DISK_SER`: Simile al precedente, ma con dati serializzati.

    ◦ `DISK_ONLY`: Salva le partizioni RDD solo su disco.

• **Checkpointing**: Il checkpointing è un meccanismo più robusto che salva un RDD su un sistema di storage affidabile, come HDFS. A differenza del caching, il checkpointing **disconnette l'RDD dal suo lineage**. Questo è utile in due scenari principali: per il recupero da guasti critici in applicazioni molto lunghe (troncando la catena di dipendenze da ricalcolare) e per riutilizzare un RDD tra diverse esecuzioni di un'applicazione Spark.

Oltre all'ottimizzazione degli RDD, Spark offre meccanismi specifici per condividere dati tra i task in modo controllato, un argomento che affronteremo nel prossimo capitolo con le variabili condivise.




# LE VARIABILI CONDIVISE IN SPARK
In un ambiente di calcolo distribuito come Spark, i task vengono eseguiti in isolamento sui nodi di lavoro del cluster. Per impostazione predefinita, ogni task opera su una copia delle variabili definite nel programma driver, e qualsiasi modifica apportata a tali copie non viene propagata né al driver né agli altri task. Questa architettura pone una sfida quando è necessario condividere informazioni tra i task. Per risolvere questo problema, Spark offre due meccanismi specializzati noti come **variabili condivise**, progettati per due specifici casi d'uso: la distribuzione efficiente di dati in sola lettura e l'aggregazione di risultati dai task al driver.

Broadcast Variables

Le **variabili broadcast** sono un meccanismo per distribuire in modo efficiente una variabile di grandi dimensioni in **sola lettura** a tutti gli executor del cluster. Esempi tipici includono una _lookup table_, un dizionario di grandi dimensioni o un modello di machine learning pre-addestrato. Senza le variabili broadcast, Spark invierebbe una copia serializzata della variabile con ogni task, generando un enorme traffico di rete.

Utilizzando una variabile broadcast, invece, Spark invia la variabile a ogni nodo executor **una sola volta**. La variabile viene quindi memorizzata in cache in forma deserializzata su quel nodo e può essere utilizzata da tutti i task in esecuzione su di esso. Questo approccio ottimizza drasticamente l'uso della rete e migliora le prestazioni. Per la gestione delle risorse, è possibile utilizzare il metodo `.unpersist()` per rimuovere la variabile dalla cache degli executor e permetterne il riutilizzo, o `.destroy()` per eliminarla definitivamente.

Accumulators

Gli **accumulatori** sono variabili che vengono utilizzate per un solo scopo: **aggregare valori** provenienti dai vari task e inviarli al programma driver. L'operazione di aggregazione deve essere **associativa e commutativa**, come un contatore o una somma.

La caratteristica fondamentale degli accumulatori è che solo il programma driver può leggere il valore finale dell'accumulatore, mentre i task possono solo "aggiungere" valori ad esso (tramite il metodo `add()`). È importante notare che gli aggiornamenti agli accumulatori avvengono solo quando viene eseguita un'**azione**. La loro affidabilità è garantita dal fatto che ogni aggiornamento viene **applicato una sola volta**. Questo assicura che, anche se un task viene rieseguito a causa di un guasto, il valore dell'accumulatore non verrà gonfiato in modo errato, garantendo la coerenza del risultato finale.

Sebbene gli RDD e le variabili condivise forniscano un modello di programmazione potente, presentano dei limiti, come la mancanza di ottimizzazioni basate sullo schema dei dati. Per superare questi problemi, Spark ha introdotto astrazioni di livello superiore, i DataFrame e i Dataset, che rappresentano la soluzione moderna per lavorare con dati strutturati.




# LAVORARE CON DATI STRUTTURATI: DATAFRAME E DATASET
Per superare i limiti degli RDD, come la mancanza di informazioni sullo schema e le limitate capacità di ottimizzazione automatica, Spark ha introdotto i **DataFrame**. Un DataFrame è un'astrazione per dati strutturati, concettualmente simile a una tabella di un database relazionale o a un data frame in R/Python. I dati sono organizzati in **colonne con un nome e un tipo**, e questa struttura ricca di metadati permette a Spark di applicare ottimizzazioni significative attraverso il suo motore di ottimizzazione delle query, il **Catalyst Optimizer**.

Creazione e Manipolazione dei DataFrame

I DataFrame possono essere creati da una vasta gamma di fonti. Le fonti di dati possono includere **file** (come CSV, JSON, AVRO), **sistemi di archiviazione** (come Hive, HDFS, RDBMS) o **RDD Spark esistenti**. Durante la lettura, è possibile specificare diverse opzioni per interpretare correttamente il file. Ad esempio, si può usare `option("header", "true")` per indicare che la prima riga del file è un'intestazione o `option("sep", "\t")` per specificare un delimitatore.

Una volta creato un DataFrame, è possibile manipolarlo utilizzando un ricco set di operazioni trasformative:

• `printSchema()`: Mostra lo schema del DataFrame, elencando i nomi delle colonne e i loro tipi di dati.

• `select("col1", "col2", ...)`: Seleziona un sottoinsieme di colonne, creando un nuovo DataFrame.

• `filter(condition)`: Filtra le righe del DataFrame in base a una condizione booleana.

• `groupBy("col")`: Raggruppa il DataFrame in base ai valori di una o più colonne, su cui è poi possibile applicare funzioni di aggregazione come `count()`, `sum()`, `avg()`, ecc.

Eseguire Query SQL sui DataFrame

Una delle funzionalità più potenti dei DataFrame è la capacità di interrogarli utilizzando il linguaggio SQL standard. Per fare ciò, è necessario prima registrare il DataFrame come una **vista temporanea** utilizzando il metodo `createOrReplaceTempView("nome_vista")`. Una volta registrata, la vista si comporta come una tabella temporanea su cui è possibile eseguire qualsiasi query SQL attraverso il comando `spark.sql("SELECT ... FROM nome_vista ...")`. Esiste anche la possibilità di creare viste temporanee globali (`createGlobalTempView`), che sono visibili tra diverse sessioni Spark.

Una Breve Introduzione ai Dataset

I **Dataset** rappresentano un'ulteriore evoluzione, combinando i vantaggi dei DataFrame con quelli degli RDD. Disponibili solo per i linguaggi della JVM (Scala e Java), i Dataset offrono le ottimizzazioni del Catalyst Optimizer (come i DataFrame) e la **sicurezza dei tipi (type safety)** in fase di compilazione (come gli RDD). Questo significa che molti errori che in un DataFrame verrebbero scoperti solo a runtime possono essere individuati dal compilatore quando si usa un Dataset. Di fatto, un DataFrame in Spark è un alias per un `Dataset[Row]`, dove `Row` è un oggetto generico non tipizzato.

La capacità di eseguire query SQL sui DataFrame è resa possibile da un componente specifico dell'ecosistema Spark, Spark SQL, che non solo abilita questa funzionalità ma offre anche un'integrazione profonda con altri sistemi, come Hive.





# SPARK SQL E INTEGRAZIONE CON HIVE
**Spark SQL** è il modulo di Apache Spark dedicato all'elaborazione di dati strutturati. Non è solo il motore che alimenta l'API DataFrame, ma è un componente completo che consente di eseguire query SQL direttamente sui dati. Questo modulo estende le funzionalità di base di Spark, offrendo un'interfaccia ad alto livello per l'analisi dei dati e un'integrazione nativa e performante con l'ecosistema Big Data esistente.

Integrazione di Spark SQL con Hive

Una caratteristica architetturale fondamentale di Spark SQL è la sua capacità di integrarsi perfettamente con un **Hive Metastore** esistente. Questa integrazione viene tipicamente realizzata posizionando il file di configurazione `hive-site.xml` di Hive nella directory di configurazione di Spark (`$SPARK_HOME/conf/`). Una volta configurato, Spark SQL può connettersi al Metastore di Hive, il che permette a Spark di:

• **Leggere e scrivere su tabelle Hive**: Le applicazioni Spark possono interrogare tabelle gestite da Hive come se fossero DataFrame nativi.

• **Eseguire query HiveQL**: Spark può eseguire le stesse query scritte per Hive, ma sfruttando il proprio motore di esecuzione ottimizzato invece di MapReduce o Tez. Questo disaccoppiamento tra il layer di metadati e il motore di esecuzione offre un percorso di migrazione semplice per carichi di lavoro Hive esistenti, consentendo di beneficiare della velocità di Spark senza riscrivere la logica delle query.

Operazioni su Database e Tabelle

Tramite Spark SQL, è possibile eseguire operazioni DDL (Data Definition Language) per gestire database e tabelle, proprio come in un sistema di database tradizionale. I comandi più comuni includono:

• **Gestione dei Database**:

    ◦ `CREATE DATABASE IF NOT EXISTS nome_db`: Crea un nuovo database.

    ◦ `DESCRIBE DATABASE nome_db`: Mostra i metadati di un database.

    ◦ `DROP DATABASE IF EXISTS nome_db`: Elimina un database.

• **Gestione delle Tabelle**:

    ◦ `CREATE TABLE nome_tabella (col1 TIPO, col2 TIPO, ...)`: Crea una nuova tabella, specificandone lo schema.

Caricamento dei Dati e Uso delle UDF

Spark SQL fornisce comandi per popolare le tabelle con dati provenienti da diverse fonti. Il comando `LOAD DATA` permette di caricare dati da file presenti sul filesystem locale (`LOCAL INPATH`) o su HDFS (`INPATH`) direttamente in una tabella.

Inoltre, proprio come in Hive, è possibile estendere le capacità di Spark SQL registrando e utilizzando **Funzioni Definite dall'Utente (UDF)**. Una UDF permette di incapsulare logica personalizzata (scritta in Python, Java o Scala) e di invocarla direttamente all'interno di una query SQL. Questo consente di integrare algoritmi complessi o regole di business specifiche nel processo di analisi dei dati in modo trasparente.

Riflessione Finale

In conclusione, Apache Spark, attraverso le sue diverse astrazioni — partendo dagli RDD a basso livello, passando per i potenti DataFrame e arrivando al modulo completo di Spark SQL — fornisce un framework unificato, versatile e performante per l'elaborazione dei Big Data. Copre un'ampia gamma di casi d'uso, dall'elaborazione dati complessa e personalizzata alla moderna analisi dati basata su SQL, confermandosi come uno strumento indispensabile nell'arsenale di ogni professionista del dato.
