# INTRODUZIONE AD AWS CLOUDFORMATION
Nel cloud computing moderno, l'approccio noto come **Infrastructure as Code (IaC)** è diventato una pratica fondamentale. Possiamo paragonarlo a un progetto architettonico per un edificio: invece di costruire manualmente e sperare che tutto funzioni, si crea un progetto dettagliato—il nostro _template_—che definisce in modo dichiarativo la struttura finale. AWS CloudFormation è il servizio nativo di Amazon Web Services che agisce come la squadra di costruzione specializzata: legge il progetto e orchestra la complessa sequenza di operazioni per realizzarlo in modo affidabile, versionabile e replicabile.

AWS CloudFormation è un servizio che consente di definire, fornire e gestire un insieme di risorse AWS correlate utilizzando dei file di testo, chiamati _template_. Questi template, scritti in formato JSON o YAML, descrivono tutte le componenti dell'infrastruttura desiderata. L'approccio è puramente **dichiarativo**: invece di specificare la sequenza di comandi per _creare_ le risorse, si dichiara lo stato finale desiderato, e CloudFormation si occupa di orchestrare tutte le operazioni necessarie per raggiungerlo, gestendo dipendenze e configurazioni in modo automatico.

==NOTA: nel campo del cloud computing con il termine infrastruttura non andiamo a identificare solo le risorse hardwere ma anche quelle risorse logiche e virtuali, come ad esempio i gruppi di sicurezza per le istanze EC2==

L'integrazione con **LocalStack** estende le potenzialità di CloudFormation direttamente nell'ambiente di sviluppo locale. LocalStack emula le API di AWS, permettendo agli sviluppatori di utilizzare il servizio CloudFormation per creare e testare stack infrastrutturali senza interagire con il cloud reale e senza incorrere in costi. Questo accelera notevolmente i cicli di sviluppo e testing. Con la comprensione di questi concetti di base, possiamo ora approfondire le funzionalità specifiche che rendono CloudFormation uno strumento così potente.




# LE FUNZIONALITÀ CHIAVE DI CLOUDFORMATION
Per gestire l'infrastruttura in modo efficace, CloudFormation si basa su una serie di componenti e funzionalità fondamentali. Queste caratteristiche abilitano un approccio alla gestione dell'infrastruttura che è allo stesso tempo robusto, controllato e scalabile, trasformando un semplice file di testo in un ambiente cloud complesso e funzionante.


Template Dichiarativi

I **template** sono il cuore di CloudFormation. Si tratta di file in formato JSON o YAML che definiscono in modo dichiarativo tutte le componenti infrastrutturali (risorse) necessarie, come istanze EC2, bucket S3, database, reti e permessi. Il valore di questo approccio risiede nella loro **riutilizzabilità** e nella possibilità di sottoporli a **controllo di versione** (ad esempio, tramite Git). Questo significa che l'intera infrastruttura può essere trattata come codice.



Stack e Stack Set

Uno **stack** rappresenta una singola unità di deployment gestita da un template CloudFormation. Quando un template viene distribuito, CloudFormation crea uno stack che contiene tutte le risorse specificate. Questo stack può essere aggiornato, eliminato e gestito come un'entità unica. Per scenari più complessi, gli **Stack Set** estendono questa funzionalità, permettendo di gestire in modo centralizzato il deployment di stack su più account e regioni AWS a partire da un singolo template, una caratteristica utile in contesti aziendali che richiedono coerenza su larga scala.



Automazione e Sicurezza

CloudFormation eccelle nell'automazione e nella garanzia di deployment sicuri attraverso due meccanismi chiave:

• **Automated Resource Provisioning:** Il servizio gestisce autonomamente le dipendenze tra le risorse. Ad esempio, se un'istanza EC2 dipende da un gruppo di sicurezza, CloudFormation assicurerà che il gruppo di sicurezza venga creato prima dell'istanza.

• **Change Sets:** Prima di applicare un aggiornamento a uno stack esistente, è possibile creare un "change set". Questo permette di visualizzare in anteprima una lista dettagliata delle modifiche che verranno apportate. Ciò garantisce un processo di deployment più sicuro e controllato.




Coerenza e Affidabilità

Per mantenere l'integrità dell'infrastruttura, CloudFormation offre strumenti aggiuntivi:

• **Drift Detection:** Questa funzionalità permette di identificare le modifiche apportate manualmente alle risorse al di fuori del controllo di CloudFormation, mantenendo la coerenza tra il template e lo stato reale.

• **Rollback Automatico:** In caso di errori durante la creazione o l'aggiornamento di uno stack, CloudFormation può automaticamente ripristinare lo stato stabile precedente, garantendo che l'infrastruttura non rimanga in uno stato inconsistente.

La comprensione di queste funzionalità è essenziale per apprezzare il flusso operativo con cui CloudFormation trasforma il codice in infrastruttura, un processo che analizzeremo nel dettaglio nel prossimo capitolo.



# IL FLUSSO OPERATIVO E I VANTAGGI DI CLOUDFORMATION
Il ciclo di vita della gestione dell'infrastruttura tramite CloudFormation segue una sequenza logica di passaggi che trasformano il codice in risorse cloud attive e gestibili. Comprendere questo flusso di lavoro è fondamentale per sfruttare appieno i benefici che questo strumento offre.


Il Flusso di Lavoro di CloudFormation

Il processo operativo può essere articolato in quattro passaggi fondamentali:

1. **Creazione del Template:** Il primo passo consiste nello scrivere un template in formato JSON o YAML che descrive tutte le risorse AWS necessarie.

2. **Deployment dello Stack:** Una volta che il template è pronto, viene distribuito per creare uno **stack**. CloudFormation legge il template e si occupa del provisioning ordinato delle risorse.

3. **Gestione dello Stack:** Dopo la creazione, lo stack può essere gestito nel tempo: aggiornato, monitorato e, quando non più necessario, eliminato. L'eliminazione di uno stack comporta la rimozione di tutte le risorse associate.

4. **Utilizzo dei Change Set (Opzionale):** Per gli aggiornamenti, è buona prassi utilizzare i **Change Set** per visualizzare in anteprima le modifiche, garantendo che gli aggiornamenti siano sicuri prima di essere applicati.



I Benefici dell'Adozione di CloudFormation

L'adozione di questo flusso di lavoro porta a vantaggi significativi:

• **Consistenza e Affidabilità:** Automatizzando il provisioning, si riduce drasticamente il rischio di errore umano e si previene il "configuration drift", garantendo che gli ambienti siano sempre consistenti.

• **Ripetibilità:** I template possono essere riutilizzati per creare repliche esatte della stessa infrastruttura in ambienti diversi (es. sviluppo, test, produzione), assicurando uniformità e rapidità di setup.

• **Scalabilità:** Modificare un template per aggiungere o rimuovere risorse è un modo semplice ed efficace per scalare l'infrastruttura verso l'alto o verso il basso in base alle necessità.

• **Automazione e DevOps:** L'approccio Infrastructure as Code si integra perfettamente nelle pipeline di CI/CD. È proprio perché i template garantiscono la **ripetibilità** che possono essere integrati in modo affidabile nei flussi di automazione, diventando il cardine delle pratiche DevOps per l'infrastruttura.

• **Gestione dei Costi:** La capacità di creare e distruggere intere infrastrutture con un singolo comando è fondamentale. Ambienti temporanei possono essere creati on-demand e rimossi al termine dell'uso, evitando spese inutili.

Ora che abbiamo compreso il flusso di lavoro e i suoi vantaggi, è il momento di esaminare più da vicino la gestione pratica degli stack e i comandi specifici necessari per interagire con essi.




# GESTIONE PRATICA DEGLI STACK CLOUDFORMATION
Dalla teoria passiamo ora alla pratica. Per implementare efficacemente l'Infrastructure as Code, è essenziale comprendere come definire e manipolare gli stack CloudFormation tramite codice e comandi da riga di comando.

Definire uno Stack

Uno stack CloudFormation è una collezione di risorse AWS gestita come una singola unità, la cui definizione è contenuta in un template JSON o YAML. Ad esempio, il seguente frammento di codice JSON definisce uno stack molto semplice che contiene una sola risorsa: un bucket S3.

```
{
  "Resources": {
    "LocalBucket": {
      "Type": "AWS::S3::Bucket",
      "Properties": {
        "BucketName": "cfn-quickstart-bucket"
      }
    }
  }
}
```

In questo esempio, `Resources` contiene le definizioni, `LocalBucket` è l'identificatore logico, `Type` specifica la risorsa (`AWS::S3::Bucket`) e `Properties` ne definisce le configurazioni.

Deployment di uno Stack

Per creare uno stack da un file di template, si usa il comando `deploy` dell'AWS CLI (o `awslocal` per LocalStack).

```
awslocal cloudformation deploy --stack-name cfn-quickstart-stack \
  --template-file "./stack.json"
```

Per verificare che le risorse siano state create correttamente, si può ispezionare il servizio corrispondente. Ad esempio, per il bucket S3, il comando `list-buckets` dovrebbe mostrare il nuovo bucket nell'elenco.

```
awslocal s3api list-buckets
```

Eliminazione di uno Stack

Quando le risorse di uno stack non sono più necessarie, il comando `delete-stack` si occupa di smantellare tutte le risorse create da quello specifico stack, garantendo una pulizia completa dell'ambiente.

```
awslocal cloudformation delete-stack --stack-name cfn-quickstart-stack
```

Dopo l'eliminazione, una nuova verifica confermerà che le risorse sono state rimosse. Eseguendo di nuovo il comando `list-buckets`, l'output mostrerà che il bucket non esiste più, rendendo l'effetto del comando inequivocabile.

```
{
    "Buckets": [],
    "Owner": {
        "DisplayName": "webfile",
        "ID": "75aa57f09aa0c8caeab4f8c24e99d10f8e7faeebf76c078efc7c6caea54ba06a"
    }
}
```

Sebbene CloudFormation sia potente, per infrastrutture complesse sono nati framework di livello superiore come l'AWS Cloud Development Kit (CDK), che offre un'astrazione maggiore.




# INTRODUZIONE ALL AWS CLOUD DEVELOPMENT KIT (CDK)
L'AWS Cloud Development Kit (CDK) rappresenta un'evoluzione significativa nel paradigma dell'Infrastructure as Code (IaC). Superando i limiti dei formati statici come JSON o YAML, il CDK consente agli sviluppatori di utilizzare la potenza e l'espressività di linguaggi di programmazione familiari per definire l'infrastruttura cloud.

Il CDK è un **framework di sviluppo software open-source** progettato per semplificare la definizione dell'infrastruttura cloud su AWS. Invece di scrivere template manualmente, gli sviluppatori possono scrivere codice in linguaggi come **TypeScript/JavaScript, Python, Java, .NET e Go**. Questo codice viene poi "sintetizzato" dal CDK in un template AWS CloudFormation standard.



I "Construct": Blocchi di Costruzione del CDK

I blocchi di costruzione fondamentali del CDK sono chiamati **Construct**. Per capire la loro utilità, pensiamoli come dei mattoncini LEGO:

1. **Construct L1 (Level 1):** Sono i singoli mattoncini base. Corrispondono direttamente alle risorse CloudFormation, hanno nomi che iniziano con `Cfn` (es. `CfnBucket`) e richiedono la configurazione esplicita di ogni dettaglio. Offrono il massimo controllo, ma richiedono più lavoro.

2. **Construct L2 gli sviluppatori possono scrivere codice in linguaggi come ￼￼TypeScript/JavaScript, Python, Java, .NET e Go￼￼. Questo codice viene poi "sintetizzato" dal CDK in un template AWS CloudFormation standard.(Level 2):** Sono come un telaio d'auto pre-assemblato. Offrono astrazioni di livello superiore con valori di default sensati e metodi di convenienza che semplificano pattern comuni (es. `new s3.Bucket`), facendoti risparmiare tempo e fatica.

3. **Construct L3 (Level 3) o Patterns:** Sono come un intero set LEGO Technic per un modello specifico. Rappresentano pattern architetturali completi che combinano più servizi per fornire una soluzione funzionante "out of the box", come un'API Gateway con un backend Lambda.

La comprensione di questi concetti fondamentali è il primo passo per apprezzare i benefici pratici che il CDK offre agli sviluppatori, che esploreremo nel prossimo capitolo.

==NOTA: questi vari tipologia di construct non devono essere usate tutte contemporaneamente, ma possiamo scegliere di usare una tipologia piuttosto che un altra oppure un mix tra le varie tipologie ==



# VANTAGGI DEL CKD E L'INTEGRAZIONE CON LOCALSTACK
Gli sviluppatori scelgono l'AWS CDK perché i suoi benefici si traducono direttamente in un ciclo di sviluppo dell'infrastruttura più rapido, flessibile e sicuro. Sfruttando la logica e le astrazioni dei linguaggi di programmazione, il CDK supera molte delle limitazioni dei template statici.

Vantaggi Chiave del CDK

• **Astrazione e Facilità d'uso:** Grazie a construct di alto livello, il CDK semplifica la configurazione dell'infrastruttura, permettendo agli sviluppatori di concentrarsi sulla logica applicativa.

• **Flessibilità dell'Infrastruttura:** L'uso di linguaggi di programmazione permette di creare codice modulare e riutilizzabile, utilizzando logica complessa per generare configurazioni dinamiche.

• **Deployment più Rapido:** Il CDK si occupa della generazione automatica dei template CloudFormation, eliminando il lavoro manuale di scrittura di YAML o JSON e riducendo gli errori.

• **Best Practice Integrate:** Molti construct di alto livello sono pre-configurati per seguire le best practice di sicurezza e affidabilità di AWS, aiutando a costruire infrastrutture robuste fin dall'inizio.

`cdklocal`: Sviluppo Locale con LocalStack

Per portare questi vantaggi nell'ambiente di sviluppo locale, è stato creato lo strumento **cdklocal**. Si tratta di un sottile _wrapper_ per la command-line interface (CLI) standard del CDK, che ne reindirizza le chiamate verso le API di LocalStack anziché verso il cloud AWS reale.

Lo scopo principale di `cdklocal` è abilitare lo **sviluppo e il testing locale delle applicazioni CDK**, permettendo di iterare rapidamente sull'infrastruttura senza incorrere in costi cloud. Lo strumento è disponibile come libreria `npm` e deve essere installato globalmente.

Con la conoscenza dei vantaggi del CDK e dello strumento `cdklocal`, siamo pronti a esplorare la struttura tipica di un progetto CDK e il flusso di lavoro necessario per il suo deployment.




# STRUTTURA E COMANDI DI UN PROGETTO CDK

Conoscere la struttura standard di un progetto CDK e la sequenza di comandi per il deployment è fondamentale. Questa conoscenza fornisce una base solida per sviluppare e gestire applicazioni infrastrutturali complesse e manutenibili.

Struttura Tipica di un'Applicazione CDK

Un progetto CDK inizializzato segue una struttura di directory convenzionale che separa le responsabilità.

```
my-cdk-app/
├── bin/
│   └── my-cdk-app.js
├── lib/
│   └── my-cdk-app-stack.js
├── cdk.json
├── package.json
...
```

Le componenti principali sono:

• **La directory** **bin****:** Contiene il punto di ingresso. È qui che **vengono istanziati gli stack** da distribuire.

• **La directory** **lib****:** Qui **viene definita la logica degli stack e delle risorse AWS**, dichiarando e configurando i vari componenti.

• **Il file** **cdk.json****:** Specifica la configurazione per l'applicazione CDK, incluso il comando per eseguirla.

• **Il file** **package.json****:** Gestisce le dipendenze e gli script del progetto.



Flusso di Comandi per il Deployment

Il deployment di un'applicazione CDK su LocalStack tramite `cdklocal` segue una sequenza di tre comandi base:

1. **Inizializzazione:** Si crea una directory per il progetto, ci si sposta al suo interno e si esegue il comando `cdklocal init sample-app --language=javascript`. Questo crea lo scheletro di un nuovo progetto CDK (`sample-app` è un placeholder per il nome del servizio).

2. **Bootstrap:** Il comando `cdklocal bootstrap` è un passaggio preliminare che prepara l'ambiente di destinazione (in questo caso LocalStack) creando le risorse di supporto necessarie al CDK.

3. **Deployment:** Infine, il comando `cdklocal deploy` sintetizza il codice CDK in un template CloudFormation e lo distribuisce, creando lo stack e tutte le risorse definite al suo interno.

Questa sequenza copre il ciclo di vita iniziale. Per una gestione efficace, è utile analizzare più approfonditamente lo scopo dei comandi più importanti del CDK.




# ANALISI DEI COMANDI FONDAMENTALI DEL CDK
La padronanza dei comandi principali del CDK — `bootstrap`, `synth`, `deploy` e `destroy` — è essenziale per gestire efficacemente l'intero ciclo di vita dell'infrastruttura come codice, dalla preparazione dell'ambiente alla sua completa dismissione.

`cdklocal bootstrap`

Questo comando prepara l'ambiente AWS (o LocalStack) a ricevere deployment dal CDK. L'operazione, da eseguire una sola volta per ambiente, crea uno stack speciale chiamato **CDKToolkit**. Questo stack contiene risorse di supporto, tra cui un **bucket S3** per lo staging di asset (come codice Lambda) e i **ruoli IAM** che concedono al CDK i permessi per operare. È necessario eseguirlo al primo deployment o quando si iniziano a usare asset.

`cdklocal synth`

Il comando `synth` (abbreviazione di "synthesize") esegue il codice dell'applicazione CDK e genera il template CloudFormation corrispondente, mostrandolo a schermo. Non esegue alcun deployment, ma si limita a tradurre il codice nell'artefatto che verrà effettivamente utilizzato da CloudFormation. Questo è estremamente utile per visualizzare in anteprima, validare e revisionare le modifiche all'infrastruttura, specialmente in flussi di lavoro CI/CD.

`cdklocal deploy` e `destroy`

Il comando `cdklocal deploy` è il cuore del processo. Implicitamente, esegue prima un'operazione di `synth` per generare il template, e poi lo invia al servizio CloudFormation per il provisioning dello stack e di tutte le sue risorse. A causa di una limitazione corrente in LocalStack, l'aggiornamento di uno stack esistente potrebbe fallire; in tal caso, la soluzione raccomandata è una sequenza di `destroy` e `deploy` per garantire una creazione pulita.

Il comando `cdklocal destroy` compie l'azione opposta: rimuove lo stack specificato e tutte, e sole, le risorse che esso gestisce. La sua precisione è un vantaggio critico rispetto a una rimozione manuale, che è soggetta a errori e spesso conduce a risorse "orfane" e costi non necessari, collegandosi direttamente al beneficio della gestione dei costi.

La conoscenza di questi comandi ci permette ora di passare a un esempio pratico che li metta in opera.




# ESEMPIO PRATICO: CREARE UNA FUNZIONE LAMBDA CON CDK
Questo capitolo finale sintetizza i concetti appresi, mostrando come l'AWS CDK venga utilizzato concretamente per definire una risorsa di calcolo, una funzione Lambda, e per configurare la sua interazione con una coda SQS in modo interamente programmatico.

Creazione di una Funzione Lambda

Definire una funzione Lambda con il CDK è un processo intuitivo che sfrutta i construct di alto livello. Il codice seguente mostra come creare una nuova funzione, specificando le sue proprietà essenziali:

```
const fn = new lambda.Function(this, 'MyFunction', {
    code: lambda.Code.fromAsset('lambdas'),
    runtime: lambda.Runtime.NODEJS_18_X,
    handler: 'lambda_sqs.lambda_handler',
    functionName: 'process-event'
});
```

In questo frammento, `new lambda.Function` istanzia la risorsa. `Code.fromAsset('lambdas')` indica al CDK di prendere il codice sorgente da una directory locale e gestirlo come un asset. `runtime` e `handler` specificano l'ambiente di esecuzione e il punto di ingresso.

Configurazione di una SQS Queue come Event Source

Uno dei maggiori vantaggi del CDK è la facilità con cui si possono collegare le risorse. Per configurare una coda SQS come trigger che invoca la Lambda ogni volta che riceve un messaggio, è sufficiente usare un metodo di convenienza fornito dal construct:

```
fn.addEventSource(new eventsources.SqsEventSource(queue));
```

Questa singola riga di codice astrae tutta la complessità sottostante: il CDK si occuperà automaticamente di creare il mapping della sorgente di eventi e di aggiungere le necessarie permissioni IAM al ruolo della funzione.

Questo percorso, dalla teoria di CloudFormation alla pratica del CDK, dimostra la potente sinergia tra questi strumenti. CloudFormation fornisce il fondamento dichiarativo e robusto; il CDK offre l'espressività e l'astrazione di un vero linguaggio di programmazione; e LocalStack crea un ambiente locale, sicuro e privo di costi per l'iterazione rapida. Insieme, ridefiniscono il flusso di lavoro moderno per lo sviluppo cloud, trasformando l'infrastruttura in un'estensione naturale e gestibile del ciclo di vita del software.
