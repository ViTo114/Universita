# INTRODUZIONE AD AWS APPSYNC E AL SUO UTILIZZO CON LOCALSTACK
Nello sviluppo moderno di applicazioni cloud, i servizi API gestiti rivestono un'importanza cruciale per orchestrare la comunicazione tra client e servizi di backend. In questo contesto, **AWS AppSync** si afferma come una soluzione strategica, offrendo un servizio serverless per la creazione di API GraphQL. Il suo ruolo è quello di semplificare radicalmente l'accesso e la manipolazione dei dati, aggregandoli in modo efficiente da una molteplicità di fonti.

Che cos'è AWS AppSync?

AWS AppSync è un servizio completamente gestito offerto da Amazon Web Services, progettato per consentire agli sviluppatori di costruire API GraphQL flessibili e scalabili. Il suo scopo principale è quello di fornire un unico endpoint attraverso il quale le applicazioni client possono interrogare database, microservizi e altre API. Una delle sue caratteristiche distintive, e un concetto fondamentale da comprendere, è la capacità di definire in modo dichiarativo i modelli di dati e la logica di business, astraendo gran parte della complessità legata alla gestione dell'infrastruttura.

AppSync può interfacciarsi con diverse tipologie di sorgenti dati (_Data Sources_), tra cui:

• **Servizi AWS**, come il database NoSQL Amazon DynamoDB.

• **Database relazionali**.

• **Sorgenti dati personalizzate**, che permettono integrazioni con qualsiasi backend tramite funzioni Lambda.

Il Valore dell'Integrazione con LocalStack

L'integrazione tra AWS AppSync e **LocalStack** rappresenta un enorme vantaggio per il ciclo di vita dello sviluppo software. LocalStack emula i servizi cloud di AWS in un ambiente locale, permettendo di sviluppare e testare le API AppSync senza la necessità di una connessione a Internet e senza incorrere nei costi associati all'utilizzo dell'infrastruttura cloud reale. Questo approccio è fondamentale per implementare un flusso di lavoro agile ed efficiente per le applicazioni basate su GraphQL.

I vantaggi strategici di questa combinazione sono evidenti:

• **Sviluppo Offline:** Permette di continuare a lavorare sulle API anche in assenza di connettività.

• **Test a Costo Zero:** Facilita l'esecuzione di test funzionali e di integrazione in modo economico.

• **Ciclo di Sviluppo Efficiente:** Accelera i cicli di feedback, consentendo agli sviluppatori di validare le modifiche istantaneamente.

Ora che abbiamo compreso i concetti fondamentali, è tempo di passare dalla teoria alla pratica. Il primo passo concreto consiste nel configurare la sorgente dati che la nostra API utilizzerà: un database DynamoDB.



# CONFIGURAZIONE DELL'AMBIENTE: CREAZIONE DELLA TABAELLA DYNAMODB
Prima di poter definire la struttura di un'API, è essenziale predisporre la sorgente dati che essa andrà a interrogare. Nel nostro caso pratico, utilizzeremo **Amazon DynamoDB**, un database NoSQL ad alte prestazioni. Grazie a LocalStack, possiamo creare e gestire una tabella DynamoDB interamente nel nostro ambiente di sviluppo locale.

Il processo inizia con la creazione della tabella tramite l'interfaccia a riga di comando (CLI) di AWS configurata per LocalStack. Utilizzeremo l'API `CreateTable` attraverso il comando `awslocal dynamodb create-table`. Analizziamo i parametri chiave di questo comando:

```
awslocal dynamodb create-table \
  --table-name DynamoDBNotesTable \
  --attribute-definitions AttributeName=NoteId,AttributeType=S \
  --key-schema AttributeName=NoteId,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST
```

• `--table-name DynamoDBNotesTable`: Assegna un nome univoco alla nostra tabella.

• `--attribute-definitions`: Definisce gli attributi che comporranno la chiave primaria. In questo caso, specifichiamo un attributo `NoteId` di tipo stringa (`S`).

• `--key-schema`: Specifica come viene composta la chiave primaria. Qui definiamo `NoteId` come chiave di partizione (`HASH`).

• `--billing-mode PAY_PER_REQUEST`: Imposta il modello di fatturazione. Sebbene i costi non siano un fattore in LocalStack, l'adozione di impostazioni simili a quelle di produzione è una _best practice_. `PAY_PER_REQUEST` è un pilastro del paradigma serverless, in cui si paga solo per l'uso effettivo, e mantenere questa coerenza tra l'ambiente locale e quello cloud semplifica la transizione del codice.

Una volta eseguito il comando, è buona prassi verificare che la tabella sia stata creata correttamente. Possiamo farlo utilizzando l'API `ListTables` con il seguente comando:

```
awslocal dynamodb list-tables
```

Se l'operazione è andata a buon fine, l'output JSON confermerà la presenza della tabella:

```
{
  "TableNames": [
    "DynamoDBNotesTable"
  ]
}
```

Con la nostra sorgente dati pronta, il passo successivo è creare l'infrastruttura dell'API GraphQL che interagirà con la tabella appena creata.





# CREAZIONE DELL'API GRAPHQL E DELLE CHIAVI DI ACCESSO
L'API GraphQL funge da punto di ingresso (_entry point_) per le applicazioni client, esponendo i dati in modo controllato e sicuro. La sua creazione è il passaggio fondamentale del nostro processo e richiede la configurazione di un meccanismo di autenticazione per proteggere l'accesso alle risorse.

Il primo passo è creare l'API stessa utilizzando il comando `awslocal appsync create-graphql-api`, che invoca l'API `CreateGraphqlApi` di AppSync.

```
awslocal appsync create-graphql-api --name NotesApi --authentication-type API_KEY
```

• `--name NotesApi`: Assegna un nome descrittivo alla nostra API.

• `--authentication-type API_KEY`: Specifica che l'autenticazione avverrà tramite una chiave API. Questa scelta è strategica per gli ambienti di sviluppo e test, poiché offre un meccanismo di controllo degli accessi semplice da implementare e gestire, senza la complessità di sistemi più robusti come OAuth o IAM, che sono invece più adatti per la produzione.

**Nota importante:** La risposta JSON a questo comando conterrà un valore `apiId`. È fondamentale prendere nota di questo identificativo, poiché sarà necessario in tutti i passaggi successivi per riferirsi a questa specifica API.

Successivamente, per consentire l'accesso all'API, dobbiamo generare una chiave di autenticazione. Questo viene fatto tramite il comando `awslocal appsync create-api-key`, che richiede l'`api-id` ottenuto in precedenza.

```
awslocal appsync create-api-key --api-id <API-ID>
```

**Nota importante:** Anche in questo caso, la risposta JSON conterrà un oggetto `apiKey`. È cruciale salvare il valore del campo `id` al suo interno (riferito come `apiKey.id`), in quanto verrà utilizzato per autenticare le richieste inviate all'API.

A questo punto, abbiamo creato una struttura API vuota e sicura. Il prossimo passo è definirne le capacità e la struttura dei dati attraverso uno schema GraphQL.





# DEFINIZIONE DELLA STRUTTURA DEI DATI: LO SCHEMA GRAPHQL
Lo schema è il cuore di un'API GraphQL. Agisce come un contratto rigoroso tra il client e il server, definendo in modo esplicito i tipi di dati disponibili, le operazioni per leggerli (query) e quelle per modificarli (mutazioni). È essenziale comprendere che ogni richiesta del client viene validata rispetto a questo schema, garantendo prevedibilità e robustezza all'intero sistema.

Analizziamo in dettaglio la struttura dello schema `schema.graphql` che definisce la nostra API per la gestione di note.

```
type Note {
  NoteId: ID!
  title: String
  content: String
}

type PaginatedNotes {
  notes: [Note!]!
}

type Query {
  allNotes(limit: Int): PaginatedNotes!
  getNote(NoteId: ID!): Note
}

type Mutation {
  saveNote(NoteId: ID!, title: String!, content: String!): Note
  deleteNote(NoteId: ID!): Note
}

type Schema {
  query: Query
  mutation: Mutation
}
```

• **type Note**: Rappresenta la nostra entità principale. È composta da tre campi: `NoteId` (un identificativo unico, obbligatorio), `title` e `content` (entrambi di tipo stringa).

• **type PaginatedNotes**: Funge da contenitore per restituire una lista di note, facilitando l'estensione futura con metadati di paginazione.

• **type Query**: Definisce le operazioni di lettura.

    ◦ `allNotes`: Recupera tutte le note, accettando un parametro opzionale `limit` per limitare il numero di risultati.

    ◦ `getNote`: Recupera una singola nota specificandone l' `NoteId`.

• **type Mutation**: Definisce le operazioni di scrittura.

    ◦ `saveNote`: Permette di creare o aggiornare una nota.

    ◦ `deleteNote`: Consente di eliminare una nota esistente.

• **type Schema**: È il tipo radice che aggrega tutte le query e le mutazioni definite, rendendole disponibili attraverso l'API.

In sintesi, questo schema modella un dominio di dati ben definito ("le note") e ne espone le capacità operative attraverso l'interfaccia standardizzata di GraphQL, separando nettamente la definizione dell'API dalla sua implementazione sottostante. Ora che lo schema è stato definito, dobbiamo caricarlo su AppSync e collegarlo alla nostra tabella DynamoDB.




# COLLEGAMENTO DELL'API ALLA SORGENTE DATI
Uno schema GraphQL, da solo, è solo una definizione astratta. Per renderlo funzionante, è necessario istruire AppSync su _come_ e _dove_ trovare i dati che lo schema descrive. È essenziale comprendere la distinzione tra lo schema logico (la nostra API) e la sorgente dati fisica (DynamoDB). Questo collegamento avviene in due passaggi fondamentali: il caricamento dello schema nell'API e la configurazione della sorgente dati.

**1. Caricamento dello Schema**

Per prima cosa, carichiamo il file `schema.graphql` nella nostra API AppSync utilizzando il comando `awslocal appsync start-schema-creation`, che invoca l'API `StartSchemaCreation`.

```
awslocal appsync start-schema-creation \
  --api-id <API-ID> \
  --definition file://schema.graphql
```

• `--api-id`: Specifica l'identificativo dell'API creata in precedenza.

• `--definition file://schema.graphql`: Indica il percorso del file di schema da caricare.

**2. Creazione della Sorgente Dati**

Successivamente, dobbiamo dire ad AppSync di connettersi alla nostra tabella DynamoDB. Creiamo una "Data Source" con il comando `awslocal appsync create-data-source`.

```
awslocal appsync create-data-source --name AppSyncDB \
  --api-id <API-ID> \
  --type AMAZON_DYNAMODB \
  --dynamodb-config tableName=DynamoDBNotesTable,awsRegion=us-east-1
```

Analizziamo i parametri di questo comando:

• `--name AppSyncDB`: Assegna un nome logico alla sorgente dati all'interno di AppSync.

• `--api-id`: Identifica l'API a cui questa sorgente dati sarà associata.

• `--type AMAZON_DYNAMODB`: Specifica che la nostra sorgente dati è una tabella DynamoDB.

• `--dynamodb-config`: Fornisce i dettagli di configurazione, come il `tableName` e la `awsRegion`. La specifica della regione (`awsRegion=us-east-1`), anche in un ambiente locale, è un punto didattico cruciale. LocalStack emula le API AWS con alta fedeltà e molti SDK e strumenti AWS richiedono che una regione sia specificata come parte della firma della richiesta, anche quando il servizio è in esecuzione su `localhost`.

Ora che l'API è a conoscenza sia della struttura dei dati (schema) sia della loro ubicazione fisica (data source), il passo successivo è definire la logica che mappa ogni operazione dello schema a un'azione concreta sul database. Questo è il ruolo dei resolver.





# IMPLEMENTAZIONE DELLA LOGICA DI BUSINESS: I RESOLVER
Un concetto fondamentale in AppSync è il **resolver**. Più che un semplice "ponte", un resolver è un componente logico che funge da strato di disaccoppiamento tra lo schema GraphQL rivolto al client e la sorgente dati di backend. È il "motore" dell'API: quando un client richiede un campo come `allNotes`, il resolver associato viene eseguito per recuperare i dati. Questa architettura è ciò che rende GraphQL così flessibile: permette allo schema di evolvere indipendentemente dalla struttura della sorgente dati sottostante. Potremmo risolvere il campo `allNotes` interrogando DynamoDB oggi, un database relazionale domani o un microservizio, senza che il client debba mai essere modificato.

In questa fase, creeremo un resolver per la query `allNotes` tramite il comando `awslocal appsync create-resolver`.

```
awslocal appsync create-resolver \
  --api-id <API-ID> \
  --type-name Query \
  --field-name allNotes \
  --data-source-name AppSyncDB \
  --request-mapping-template file://allNotes_request.vtl \
  --response-mapping-template file://allNotes_response.vtl
```

Esaminiamo il significato di ogni parametro:

• `--api-id`: Identifica l'API di appartenenza.

• `--type-name Query`: Specifica che il resolver si applica a un campo definito all'interno del tipo `Query` del nostro schema.

• `--field-name allNotes`: Indica il campo specifico a cui il resolver è associato.

• `--data-source-name AppSyncDB`: Collega il resolver alla sorgente dati DynamoDB che abbiamo configurato.

• `--request-mapping-template` e `--response-mapping-template`: Questi sono i due componenti cruciali che definiscono la logica di trasformazione, facendo riferimento a file che contengono il codice per mappare la richiesta in entrata e la risposta in uscita.

A questo punto, è necessario approfondire il funzionamento di questi _mapping template_, poiché rappresentano il vero e proprio "cervello" del resolver.




# LA LOGICA DI TRADUZIONE: I MAPPING TEMPLATE VTL
Un **mapping template** è essenzialmente un traduttore. Il suo compito è convertire la richiesta GraphQL in un formato comprensibile dalla sorgente dati (un comando per DynamoDB) e, al ritorno, trasformare la risposta della sorgente dati nel formato JSON atteso dal client GraphQL. Questi template sono scritti in **Apache Velocity Template Language (VTL)**, un linguaggio che permette di inserire logica dinamica.

Il motore VTL di AppSync opera su una struttura dati primaria nota come variabile `$context`. Questa variabile contiene tutte le informazioni relative alla richiesta (argomenti, header, identità del chiamante) e, nel template di risposta, il risultato grezzo restituito dalla sorgente dati.

Request Mapping Template (Richiesta)

Il template di richiesta viene eseguito prima che la chiamata venga inoltrata alla sorgente dati. Per la nostra query `allNotes`, il template ha il seguente aspetto:

```
#set($limit = $util.defaultIfNullOrEmpty($context.arguments.limit, 10))
{
  "version": "2018-05-29",
  "operation": "Scan",
  "limit": $limit
}
```

Analizziamolo nel dettaglio:

• `#set($limit = $util.defaultIfNullOrEmpty($context.arguments.limit, 10))`: Questa direttiva VTL accede a `$context.arguments` per verificare se il client ha passato un argomento `limit`. Se assente, imposta un valore di default di `10`.

• `"version": "2018-05-29"`: Specifica la versione del template di mapping di AppSync.

• `"operation": "Scan"`: Questa è l'istruzione chiave che dice ad AppSync di eseguire un'operazione di `Scan` su DynamoDB, che legge tutti gli elementi della tabella. Per un'applicazione didattica è una scelta semplice, ma in produzione va usata con cautela su tabelle di grandi dimensioni per le sue implicazioni su costi e performance.

Response Mapping Template (Risposta)

Il template di risposta viene eseguito dopo aver ricevuto i dati da DynamoDB e prima di inviarli al client. Il suo scopo è formattare i dati e gestire eventuali errori.

```
#if($context.result && $context.result.errorMessage)
  $util.error($context.result.errorMessage, $context.result.errorType)
#else
  {
    "notes": $util.toJson($context.result.items)
  }
#end
```

La sua logica è la seguente:

• `#if($context.result && $context.result.errorMessage)`: Controlla se l'oggetto `$context.result`, che contiene la risposta grezza da DynamoDB, presenta un messaggio di errore. In caso affermativo, usa `$util.error` per propagare l'errore al client GraphQL.

• `#else`: Se non ci sono errori, estrae la lista di elementi da `$context.result.items` e la inserisce in un oggetto JSON sotto la chiave `"notes"`. La funzione `$util.toJson` garantisce che la lista venga serializzata correttamente.

Con tutti i pezzi al loro posto — database, API, schema, resolver e template — siamo finalmente pronti per interrogare la nostra API e vederla in azione.




# INTERROGAZIONE DELL'API: ESECUZIONE DI UNA QUERY GRAPHQL
La fase finale del processo consiste nell'interagire con l'API appena costruita, simulando una richiesta da parte di un'applicazione client per recuperare i dati. L'esecuzione della query `allNotes` richiede quattro passaggi sequenziali.

**1. Preparare la Richiesta HTTP:** La prima azione è preparare una richiesta HTTP di tipo `POST`. Questa richiesta deve essere indirizzata all'endpoint URL dell'API GraphQL, che può essere recuperato dalla Web application di LocalStack.

**2. Impostare l'Autenticazione:** Poiché abbiamo configurato l'API per utilizzare l'autenticazione tramite chiave, è necessario includere l'header `x-api-key` nella richiesta. Il suo valore deve corrispondere all'ID della chiave API che abbiamo salvato durante la fase di creazione.

**3. Costruire il Corpo della Richiesta:** Il corpo (_body_) della richiesta POST deve contenere un oggetto JSON con una singola chiave, `"query"`. Il valore di questa chiave è la query GraphQL che si desidera eseguire, espressa come stringa. Per recuperare tutte le note con i loro campi, la query sarà:

```
{
  "query": "query { allNotes { notes { NoteId title content } } }"
}
```

Questa sintassi chiede all'API di eseguire la query `allNotes` e, per ogni nota nella lista restituita, di includere i campi `NoteId`, `title` e `content`.

**4. Inviare la Richiesta:** L'ultimo passo è l'invio effettivo della richiesta HTTP POST così costruita. Se tutti i passaggi precedenti sono stati eseguiti correttamente, il server AppSync emulato da LocalStack risponderà con i dati richiesti in formato JSON.

In questo percorso di studio, siamo partiti dalla definizione teorica di AWS AppSync per arrivare alla costruzione e all'interrogazione di un'API GraphQL completamente funzionante, il tutto all'interno di un ambiente di sviluppo locale gestito da LocalStack. Abbiamo configurato una sorgente dati, definito uno schema, implementato la logica di business con i resolver e, infine, eseguito una query reale. Questo processo end-to-end consolida le competenze pratiche necessarie per sviluppare, testare e gestire applicazioni serverless moderne basate su GraphQL.
